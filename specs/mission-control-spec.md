# Mission Control — Consolidated Specification

## 1. Philosophy

Mission Control is a lightweight, local-first autonomous work orchestration system for multi-agent teams. It is **deployed once**, pointed at a few **Worksources** (units of jurisdiction), and runs continuously — possibly unattended for days — coordinating AI agents through a **propose → harden → execute → verify → package** loop with no human in the inner cycle.

The scarcest resource is **operator attention**, not tokens or compute. Everything here exists to protect it:

- **Backpressure governs the production rate.** The operator can review a small, fixed number of deliverables (default **3**). When that buffer is full, the system does not pile up more work — it redirects idle capacity into **deepening** what is already queued. An unattended day is not a backlog; it is funding for refinement.
- **Leverage is structural, not requested.** You cannot simply tell an agent "do high-leverage, quality work." The role topology *enforces* it: generation is split from judgment (the Contrastive Leverage Gate), "done" is checkable (criteria written before work starts, Inv. 12), and every operator-facing artifact is decision-ready.
- **The operator sees only decision-ready Review Packets and genuine decision points (blocked tasks)** — never raw work-in-progress or correction churn.
- **Deterministic transitions.** Every state mutation follows strict mechanical rules — the control table, the gate ladder, the substrate triggers — with no judgment in the machinery. Judgment (what to propose, whether work is good, what deserves the next slot) lives only inside role sessions, and it enters the system exclusively as written state via `mc`; the deterministic layer then acts on that state. An LLM's opinion never moves the state machine directly.

## 2. Invariants (canonical — stated once, referenced by number)

1. **Single-threaded system-wide.** At most one leased pipeline role run exists across the *entire* system. Homie sessions are the explicit lease-free operator tier (§15), so they never count as pipeline runs. A single global execution lease enforces pipeline serialization. A *lease* here is a single-holder lock carrying two distinct time bounds: a renewable last-seen time kept alive by a heartbeat, so a run that goes silent is detected, and a non-renewable hard deadline that no heartbeat can extend, so a hung-but-live run is bounded too (§10); a crashed run never wedges the system. The time bounds are the whole point, which is why it is a *lease* and not a plain lock. It is a liveness/mutual-exclusion marker across ephemeral ticks, not a parallelism mechanism. Even the correction rally is serialized: a Verifier send-back is a state transition (`worked → seeded`, §6), and the Worker's next round is its own dispatched run under the same lease.
2. **`mc` is the sole writer — a kernel-enforced gate, not an advisory cooperator.** Every state mutation is one ACID SQLite transaction through `mc`, and this holds as a guarantee, not a discipline: an agent's own OS identity is denied access to the spine, and the privileged `mc` binary is the single broker permitted to open it — an agent that tries to reach the database any other way is refused by the kernel (§11.5). Agents invoke the real `mc` directly; there is no stub, no invented protocol, and no durable handoff file. They never spawn dispatch-level successors either: a spawn is split into a *decision* (dispatch logic in `mc`, inside the lock domain) and an *effect* (the host-side resident creates the session folder, wires the proxy, and starts the container — §10, §12). Read-only subagent fan-out (Inv. 14) is the lone exception and takes no lock.
3. **One action per tick, one dispatcher.** `mc dispatch` is invoked from exactly one place — the resident's timer tick (§10) — and runs only when the global execution lease is free; each tick commits exactly one action — one mutation or one claim-and-spawn. There is no handoff machinery between roles: a completing stage writes its state and releases the lease, and the next tick selects the follow-on stage from the written records. Every other `mc` verb — completions, operator actions, conversation traffic — only writes state and never dispatches; the up-to-one-tick latency this adds to stage transitions and conversation wake is accepted in a system whose cadence is minutes.
4. **Lock before dispatch.** For any spawned role, the Run row and the lease claim commit *before* the process starts. Dispatch never mutates the subject — pipeline position moves only at stage completion (Inv. 10) — so a crashed dispatch leaves no in-flight marker to clean up.
5. **Self-terminate after dispatch.** Every role and every tick exits after one mutation or one spawn. No resident agent waits for results.
6. **Canonical state is authoritative, and the spine is the message bus.** Agents derive decisions from state read via `mc`, never from chat history or files directly. The database is ground truth — including for conversation: a chat message is a spine row and a reply is a spine row; there are no mailboxes, no parked handoff files, and no message queue held in any process's memory (§15.5).
7. **Activity log is append-only.** A substrate trigger forbids `UPDATE`/`DELETE`. `actor` is the *logical originator*; the *physical writer is always `mc`*.
8. **Feed-forward topology.** Messages flow downstream only through written state (`Strategist(propose) → Editor → Worker/Strategist(initiative) → Verifier → Packager`). No upstream messaging; the only upstream influence is the bounded correction rally, which travels as durable state — a `worked → seeded` send-back plus a correction file — never as a message (Inv. 1).
9. **Producer ≠ judge, on decorrelated runtimes.** The actor that produces an artifact never grades it. Each quality gate runs in a *fresh session* on a *different runtime family* from the producer (§9 routing).
10. **Status advances only inside the transaction that completes a stage.** A task's `status` is the last stage it *completed*, written by `mc complete` together with the stage's output, atomically. Nothing ever marks a task "in progress": a crashed or reaped run simply never advanced its row, and the next tick re-selects it. This is what lets dispatch be recomputed from the records (§10) with no index to drift.
11. **A packet holds its queue slot for life.** A review packet is born only from a `packaged` task, and exactly one exists per task. Refinement and operator revise re-enter the *task* (`packaged → seeded`) while the packet stays put — still counted against the WIP cap — until an operator decision archives it. Deepening work can therefore never inflate the queue (§8).
12. **Done-ness is written before work starts.** A task's description — for an initiative, its charter — is written as checkable success criteria, and the Verifier judges by mapping each criterion to concrete evidence. The Editor rejects proposals whose done-ness cannot be checked, and its holistic plan review holds wave children to the same bar. Criteria live in prose, not typed rows; the discipline lives in the role briefs and the Verifier's rubric.
13. **Subscription-billed runtimes (selection rationale).** The runtimes were chosen because they operate on *subscription* plans (Codex ChatGPT-plan OAuth, Anthropic subscription, MiniMax Subscription key) rather than metered per-token APIs — that is *why* they were picked, not a per-dispatch check. There is no auth-assertion step; the spec assumes the configured runtimes are subscription-billed. (The one practical guard lives at runtime setup: keep metered API keys out of harness/tool envs so a silent metered fallback can't bill — see §9 Codex.)
14. **Every role may self-orchestrate; all subagents are read-only.** Any pipeline role may, inside its one leased run, orchestrate read-only subagents through its runtime's self-orchestration mechanism (§9.2); Homie may do the same inside its lease-free operator session. Subagents take no lease, write no deliverable, and never touch the spine; the top-level role remains the sole writer of its unit of work — a task needing multiple *writers* is still a scoping failure. Fan-out is depth-1 by construction: both harness families strip the spawn capability from children, so subagents cannot spawn further. Strategist(propose)'s seeding scouts remain the one fan-out that may span multiple Worksources — read-only and lease-free, under the Inv. 22 seeding-scope exception. **Orchestration is the default posture, not a granted permission** (§9.2): a role plans its unit of work as an orchestration and executes inline only by stated exception. The boundary test for any single spawn is a **compressible boundary**: a sub-problem whose full working context can be discarded once a small artifact (a brief, a verdict, a ranked list) returns. If the parent must hold the whole thing in its head to take the next step, it does not spawn.
15. **Runtime-blind above the adapter.** All provider knowledge lives in the Runtime Adapter; nothing above it — dispatch, the spine, the dashboard — branches on `runtime`.
16. **Runtime auth is not tool-visible.** Subscription credentials are visible to the harness/adapter control plane only — never to model-invoked shell/tools. A Runtime that can't enforce this split is disqualified for shell-capable roles.
17. **Everything lands as a packet.** There is no non-reviewable lane: every task that completes the pipeline ends as a review packet awaiting the operator. Infra, tooling, and bench work are ordinary proposals competing in the pool, never a side door around review. (If a non-reviewable lane ever returns, it must be pre-authorized per Worksource policy, never self-granted.)
18. **Review queue is WIP-limited.** At most `REVIEW_WIP_CAP` (default 3) unarchived review packets exist at once, across all Worksources, enforced by a substrate trigger at packet birth — not by dispatch politeness. This single cap is the only queue limit; a blocked task waits for the operator without consuming a review slot.
19. **Production is backpressure-gated.** When the queue is at cap, no new pipeline work is dispatched at all; each tick instead advances or starts refinement of what is already queued, and idles once every queued packet is saturated (§8).
20. **All agent prompts are frozen.** No role self-modifies; each is fixed scaffolding expressive only within its constraints.
21. **Time is not a dispatch input.** Wall-clock affects only leases, watchdogs, and explicit stored schedules (e.g., the Daily Console delivery time). Eligibility and dispatch order are time-invariant.
22. **Scoped jurisdiction.** Every run operates only within its Worksource's Sandbox Profile; it cannot reach another Worksource's workspace, artifacts, session files, packets, or secrets. Two read-only exceptions are explicitly declared and enumerated, both filesystem-read-only, mounting several Worksources only to read: the operator-scope Homie (§15), which is exceptional in *both* read scope and lifecycle — it takes no execution lease and lives as long as the conversation — and the Strategist(propose) **seeding-scope** pass (§3, §11.1), which is exceptional in read scope *only*: an ordinary leased pipeline run whose mount plan happens to span Worksources.
23. **Host components are native, supervised, and stateless.** The resident process — which holds the Discord connection, runs the outbox delivery loop, hosts the credential-injecting egress proxy, and fires the dispatch tick — and the dashboard are ordinary host processes under the host init system, not containers. They restart automatically on crash and return automatically at next login after logout or reboot, with nothing lost; they hold no authoritative state and run no agent logic — every durable fact lives in the spine, so a logout is only a longer gap between ticks. Dispatch runs on the resident's timer tick alone; inbound conversation traffic only writes state for the next tick to read (§10, §12).
24. **One lock domain.** The spine is a single SQLite database, and SQLite is only safe when one operating-system kernel arbitrates its locks. It therefore lives on a volume local to the container runtime's kernel — never a shared host path — and every process that opens it does so from inside that one kernel. On macOS/Windows a native host process consequently cannot open the spine directly; it invokes the real `mc` across the host→runtime boundary instead (§11.5, §12).
25. **Work is isolated in worktrees; main moves only on approval.** A standalone task runs in its own ephemeral worktree (`mc/task-<id>`) off main, never in the primary checkout. An initiative gets **one branch and one shared worktree**, cut at promotion; every wave child executes inside it, and a child's changes are committed — tagged with its task id — in the same breath as its Worker-stage completion, so the Verifier verifies a commit and the packet cites a stable range. Merging into main always requires an explicit operator approval — the packet decision for a standalone task, the arc-packet decision for an initiative — which the resident then lands mechanically on a following tick (§7). An interrupted or crashed run leaves nothing in the path of subsequent work — its partial output sits uncommitted and inert, removable host-side with no agent involvement.
26. **Every run's trace is kept forever.** Each run — pipeline role or Homie session alike — owns one trace-only session folder on the file plane, created before its container starts and mounted only into that container; the harness writes its native session record directly into it as it runs. Session folders are never swept: the folder holds nothing but the trace, and the trace is the deployment's permanent record (§9, §11.6).

## 3. Roles

Leased pipeline roles run as **fresh, ephemeral sessions**, do one unit of work, report via `mc`, and self-terminate. Generation and judgment never share a context. Every pipeline role may **self-orchestrate** that one unit of work internally — read-only fan-out under Inv. 14; mechanisms, default posture, and reference patterns in §9.2 — but the chain *between* roles never varies. Homie is the one operator-spawned conversational role: lease-free, multi-turn, and confined to the §15 operator liaison surface.

| Role | Job |
|---|---|
| **Strategist** | Runs in three explicit modes. **`propose`**: spawned when nothing else is dispatchable; reads the Worksource directive plus real workspace signal and inserts fresh proposals — ordinary tasks and initiatives alike — into the shared `proposed` pool. Its brief carries recently rejected titles as dedupe memory, so it never re-pitches what the Editor already declined. **`initiative`**: the initiative's mind at wave boundaries, dispatched when a seeded initiative is **drained** (§6.1); merges main into the initiative branch, re-reads the charter, the child ledger, and the repo as it now stands, then either composes the next wave of children (born `seeded`, reviewed by the Editor's plan review — never per-child in the proposal pool) or declares the charter satisfied with a completion report (`seeded → worked`). Operator refinement notes come back to it as ordinary waves. **`console`**: compiles the Daily Review Console (§14). Does **not** pick winners or dispatch. |
| **Editor** | Fresh session, blind to Strategist(propose)'s reasoning. Dispatched **batch-wise over the entire proposed pool**: ranks every open proposal on leverage-per-cost — a big initiative competes directly with a small sure thing — then promotes the best (`proposed → seeded`) and rejects the rest (`decision = 'rejected'`, archived, reason in prose). Promote or reject only; no defer, no merge, no rewrite power — overlap with existing work is a rejection. Must promote at least one unless every proposal duplicates existing work or is actively harmful (`mc` rejects a zero-promotion report when the ready queue is empty). Also performs the **holistic plan review** of every wave a Strategist(initiative) composes (§6.1). |
| **Worker** | Executes exactly one task to completion in its worktree (Inv. 25); captures visual evidence for UI/UX changes. Its `seeded → worked` advance commits in the same transaction that records its output — for an initiative child, in the same breath as the commit of its changes into the shared branch. |
| **Verifier** | Single reviewer (not a panel): one session, one verdict, one accountability point — internal adversarial fan-out via self-orchestration (§9.2) is evidence-gathering, never a second judge. Judges the work where it lives (Inv. 25), runs each criterion's check and the Promotion Gate Ladder (§7), and drives the bounded correction rally (`worked → seeded`, max 3). For an initiative's completion declaration it audits the criterion-to-evidence mapping **and** that the branch merges cleanly onto current main. Verifies refinement rounds like any other work — a pass is a *genuine* deepening, a fail is *churn* (§8). |
| **Packager** | From the durable record only (fresh session), renders the decision-ready packet with media embedded inline — a task packet for ordinary work, the **arc packet** (charter, waves, child ledger, criterion evidence, retirements) for an initiative. Read-only w.r.t. workspace; never re-judges. |
| **Refiner** | Converts review-queue wait time into polish. Under backpressure it reads one non-`saturated` queued **task** packet, scopes a single deepening, and re-enters the task (`packaged → seeded`); the ordinary Worker→Verifier→Packager chain carries it while the packet keeps holding its queue slot (§8). Initiative packets are refined by Strategist(initiative) instead — the notes come back as waves. |
| **Homie** | Operator-spawned conversational role (§15). A Homie session is lease-free, multi-turn, and filesystem-read-only across the operator-scope profile; it may issue only the allowed operator verbs through its agent-scoped `mc` capability (§15.3), writes no deliverable, and sits outside producer/judge accounting. |

**Contrastive Leverage Gate.** Strategist(propose) generates; the Editor judges, in a separate session blind to Strategist(propose)'s reasoning — and it judges *contrastively*, ranking the whole pool rather than scoring any idea in isolation, because a lone idea always looks worthy. Human-seeded work (`origin:user`) enters the same pool; filing it at priority **P-1** partitions it ahead of all backlog in dispatch (§10), so an urgent operator request reaches Editor review on the next free tick instead of waiting its turn.

## 4. Canonical State (the spine)

Structured state lives in **one embedded SQLite database**, written exclusively by `mc`, one transaction per mutation. Large media/traces stay as files. Zero database infrastructure: an embedded file, no database server. There is no markdown copy and no Git versioning of state.

**One lock domain (Inv. 24).** SQLite is only safe when one operating-system kernel arbitrates its locks, so the spine lives on a **runtime-local named volume** inside the container runtime's kernel, never on a shared host path. On Linux that kernel is the host's own; on macOS or Windows it is the single container-runtime VM. The corollary defines the whole topology: a native host process cannot open the spine directly on macOS — the bytes live inside the VM, in a different kernel, and there is no host path to the file — so host components reach it by invoking the real `mc` inside the lock domain (§11.5, §12). This is not a permissions problem; making the spine reachable to the host as a plain local file would split it across two kernels and corrupt it.

**The dispatch substrate is two tables.** `tasks` carries everything that flows through the pipeline — proposals, ordinary tasks, initiatives, and wave children are all rows in it — and `review_packets` carries the operator-facing queue. There is no activations table and no dispatch index: dispatch is recomputed from these records on every tick (§10). A row's `status` is the last stage it *completed* (Inv. 10), so the records always encode who is owed what; a crashed agent never advanced its task, and the same dispatch query re-selects it after the lease reaps. Initiatives are not a separate table — a row with `scope = 'initiative'` shares the proposal pool, the Editor review cohort, and the state machine with everything else (§5, §6).

**Supporting tables** (infrastructure, not work state): `worksources`, `sandbox_profiles`, `runs`, `lock` (singleton), `activity`, plus the communication-surface state of §15: `homie_sessions` and `homie_bindings` (the Homie session registry, §15.4), the **conversation records** — every inbound message and reply is an append-only spine row with durable conversation identity and stable order, alongside the pending-inbound claim state the Homie runner consumes (§15.5) — and `outbox` (the system→operator delivery queue, §15.5).

**The file plane (not queried by the dispatcher):** two kinds of bytes do not belong in a database and do not need the lock domain, so they live on host-local storage beside the spine, read directly by the native host components. First, **per-run session folders** `sessions/<run_id>/`: every run — pipeline role or Homie session alike — owns one, created before its container starts and mounted only into that container, at the path the harness writes its session file to. The harness's native record (`native.jsonl` — Codex's `rollout-*.jsonl`, the Claude SDK's `<uuid>.jsonl`; §9 — the harness's own record, never a Mission-Control-built trace) therefore lands directly on durable storage as it is written, with no copy-out step, and is **kept forever, for both tiers** (Inv. 26): the folder holds nothing but the trace, and there is no retention sweep. Second, **message attachments**: an image, document, or audio clip the operator sends or a Homie returns is written to the file plane, and its conversation row carries only a reference (§15.5). The rest of the file surface: `workflows/<run_id>-plan.js` (captured dynamic-workflow plan scripts, a diffable audit artifact; §9.2), `outputs/`, rendered packet HTML, `corrections/mc-<task_id>-corrections<n>` (Worker↔Verifier rally feedback), `revisions/<task_id>-OP-REVISION.md` (operator revise / Refiner deepening notes, carried into the re-entered pipeline's run briefs), `context/<task_id>-STEER.md`, and `routing.md` (static operator config). `run.json` — the run's trusted launch envelope — is deliberately *not* on this list: it is ephemeral launch input, mounted read-only and separately from the session folder, and removed with the container (§11.3, §11.5); the session folder contains neither launch input, terminal reports, heartbeats, mailboxes, nor scratch. These files are records and payloads, never a message channel — the bus is still the spine alone (Inv. 6).

**Write boundaries are disjoint:** `mc` owns the spine (kernel-enforced, Inv. 2); the harness writes its own native session file into its mounted session folder, with the **Runtime Adapter** arranging that write and owning the captured workflow plan scripts — the adapter never touches the spine (Inv. 2, 15).

**Make-impossible-states-impossible at the substrate.** A bug in `mc` cannot write an illegal state, because the storage rejects it:

| Rule | Enforcement |
|---|---|
| Only legal status transitions (the §6 machine, both scopes) | `BEFORE UPDATE` trigger vs. the legal-transition list |
| Correction rally ≤ 3 | `CHECK (correction_count BETWEEN 0 AND 3)` |
| Blocked requires a reason | `CHECK (blocked = 0 OR blocked_reason IS NOT NULL)` |
| No archive without a decision; decision and timestamp travel together | paired `CHECK`s |
| Approval only of packaged work | `CHECK (decision <> 'approved' OR status = 'packaged')` |
| No nested initiatives | `CHECK (initiative_id IS NULL OR scope = 'task')` |
| Wave children born `seeded`, into a live seeded initiative only | `children_birth_rules` trigger |
| No done-declaration with open children (strict drain) | `initiatives_declare_requires_drained` trigger |
| One blocked child blocks the initiative; auto-clears when the last clears | propagation trigger pair |
| Archiving an initiative cancels its open children, and their packets with them | cascade trigger |
| A packet is born from a `packaged` task only | `packets_require_packaged` trigger |
| Review queue ≤ `REVIEW_WIP_CAP` | `packets_wip_cap` trigger aborts the birth of a fourth packet |
| Saturation is computed, never hand-set | `packets_saturate` trigger at `refine_streak ≥ 3` |
| Activity append-only | `BEFORE UPDATE/DELETE` raises `ABORT` |

Whether a charter's criteria actually *cover* the goal remains the Editor's semantic judgment (Inv. 12); the substrate guarantees every stored state is a legal one.

## 5. Core Entities

### Worksource
A unit of jurisdiction: workspace, scoped credentials, tools, and a standing directive.
`id` · `title` · `kind` (`repo`|`personal`|`transient`) · `jurisdiction` · `sandbox_profile` · `directive` · `seeding_mode` (`propose-only`|`auto`) · `status` (`active`|`paused`|`archived`).

Two intake paths: **Strategist-seeding** (gated by the Leverage Gate + `seeding_mode`) and **Human-seeding** (`origin:user`). Transient Worksources self-archive when their initiative terminates.

**A mutable Worksource is a git repository.** Any Worksource whose workspace Mission Control may mutate is a git repo — code, vault, or otherwise; the isolation, review, and landing story is git's in every case. `mc worksource add` on a non-repo offers `git init`; if the operator declines, the Worksource is registered **read-only**: workers may read the workspace, deliverables land on the artifact plane only, and no workspace-mutating task can exist against it. There is no non-git mutation path. Mission Control operates on **committed state**: task branches are cut from the target ref's last commit, and the operator's uncommitted edits are invisible to the pipeline until committed — identically for code repos and vaults, with no per-workspace-type behavior and no automatic commits of operator work.

### Sandbox Profile
The concrete confinement for every run in a Worksource (Inv. 16, 22). Worksource `jurisdiction` says what is *allowed*; the profile says how it's *enforced*. **Every run is containerized — there is no host-execution mode.**
`workspace_root` · `artifact_roots` · `readonly_mounts` · `denied_paths` (includes other Worksources + host credential dirs) · `tool_home_dir` (synthetic HOME inside the container, never the operator's) · `runtime_control_dir` (harness auth/session files; denied to tools) · `harness_env_policy` · `tool_env_policy` (runtime auth env stripped) · `egress_policy` (`none`|`allowlist`|`open+audit`) · `network_allow` (raw-TCP host:port allowlist, §11.4) · `tool_allowlist` · `runtime_auth_delivery` (`gateway`|`materialized`).

`egress_policy` and `network_allow` govern two different planes and are not interchangeable: `egress_policy` controls HTTP egress through the proxy and is addressed by domain, while `network_allow` controls raw, non-proxy TCP egress and is addressed by host:port. The proxy can only enforce the former — `libpq`, `ssh`, and similar clients never consult `HTTPS_PROXY` — so anything that opens a raw socket (a Postgres connection, an SSH session) is expressed here rather than in `egress_policy`. The field is a list of host:port rules, each a DNS name or CIDR paired with a port; it **defaults to empty, which is the current deny-all-non-proxy behavior unchanged**. Opening, say, `db.xyz.supabase.co:5432` is one entry, and it also requires letting DNS resolve that host. The full two-plane model and its rulings live in §11.4.

How the container enforces the profile — images, the mount plan, the credential gateway, and the boundary protocol — is specified in §11.

### Secrets — no secrets subsystem in v1
There is no secrets store, no keychain integration, and no per-dispatch secret plan. Two kinds of secret material exist, handled differently:

- **Runtime auth (model provider credentials) is the only secret material Mission Control manages.** It is captured at onboarding (§17.3) directly into the host-side homes the resident already owns: the egress gateway's injection table for static-token bindings (`minimax`) and the canonical `runtime_control_dir` for the OAuth bindings (`codex`, `claude`) — the §11.4 delivery tiering. The resident hosts the proxy and effects every spawn, so these values are born and held exactly where they are used; there is no resolution step.
- **Worksource tool secrets (deploy keys, DB connection strings, service tokens) are not Mission Control records.** The operator keeps them where they already live — typically a `.env` in the workspace — and agents read them in plaintext like any other workspace file. Scoping is the mount plan: a container mounts only its own Worksource and `denied_paths` excludes all others, so no run can read another Worksource's secrets. There is no per-role or per-run secret assignment.

`mc worksource add` must deliver a **connect-time advisory** — required machinery, not documentation: agents can read every secret in this workspace and can send them anywhere `egress_policy` permits; use scoped, expiring, revocable credentials (limited DB roles, fine-grained PATs), keep the file gitignored, and set `egress_policy: allowlist` on secret-heavy Worksources. The default `egress_policy` for a new Worksource is `open+audit` (raw TCP stays `network_allow`-enumerated, §11.4); tightening to `allowlist` is the operator's per-Worksource dial.

The gateway's credential-injection rail (§11.4) is the reserved upgrade path — host/path-matched injection of HTTP-shaped tool secrets with per-Worksource assignment — and nothing in v1 may be designed in a way that forecloses it.

### Task
The one work table — proposals, ordinary tasks, initiatives, and wave children are all rows here; `scope` and `status` say which.
`id` · `title` · `description` (for an initiative: the **charter**, written as checkable success criteria, Inv. 12) · `scope` (`task`|`initiative`) · `initiative_id?` (set on wave children only; the substrate forbids nesting) · `priority` (−1 = expedited, 0–3 = P0–P3; lower wins) · `created_at` · `status` (§6 — the last stage *completed*) · `stage_entered_at` · `correction_count` (rally budget, ≤ 3) · `blocked` + `blocked_reason` (a dispatchability **flag**, deliberately not a status — blocking must never destroy pipeline position) · `dispatch_retries` (infra-death budget; decremented only by the reaper, §10) · `decision` (`approved`|`rejected`|`cancelled`) + `decided_at` · `archived` · `origin` (`user`|`autonomous`) · `worksource` · `stage_rank` (generated from `status`: how far into the pipeline the row is, for furthest-first dispatch, §10; `packaged` ranks 0 — in the review queue, out of pipeline dispatch).

A **proposal** is simply a task at `status = 'proposed'` — there is no separate proposals table, and the blind cross-session handoff the Leverage Gate depends on (Inv. 9) is just the Editor reading the pool back from the spine (Inv. 6). The **leverage ledger** is equally free: rejected rows keep their titles and reasons (`decision = 'rejected'`, archived), and the twenty most recent feed Strategist(propose)'s next brief as dedupe memory (§10).

### Initiative
A large, high-leverage scope of work — a tasks row with `scope = 'initiative'`. It is proposed, reviewed by the Editor, prioritized, blocked, and archived exactly like a task; only the stage *meanings* shift (§6.1). Its `description` is the **charter**: success criteria checkable enough that a later session can declare, and a Verifier can audit, that the scope is satisfied. Work happens through **waves** of child tasks (`initiative_id` set, born `seeded`); a wave is not a record, just the set of currently-open children. While children are open the initiative is **parked** — owed nothing, invisible to dispatch; when the last child is operator-decided it is **drained** and owed planning (§6.1). No nested initiatives: a child is always `scope = 'task'`.

### Review Packet
The operator-facing deliverable for one packaged task — exactly one row per task (`task_id` is both primary key and foreign key).
`task_id` · `render_path` · `thesis` · `refine_streak` (consecutive failed refinements; a genuine deepening resets it) · `saturated` (computed at `refine_streak ≥ 3`, never hand-set) · `archived` · `created_at`.

A packet is born only from a `packaged` task and holds its queue slot until an operator decision archives it — including across refinement round-trips, when the task re-enters the pipeline while the packet stays put (Inv. 11); re-packaging re-renders it in place. A wave child's packet diffs against the **initiative branch**, not main. An initiative's own packet is the **arc packet** — charter, waves, child ledger, criterion evidence — and approving it writes the landing-pending mark; a following tick lands the merge to main, archives, and deletes the worktree (§6.1, §7).

### Operator attention (packets + blocked tasks)
There is no escalations table. The operator's attention surface is two things: **review packets** (decisions on finished work, WIP-capped per Inv. 18) and **blocked tasks** (decisions only the operator can make, flagged in place). A run that cannot proceed reports `blocked` with the question in `blocked_reason`; the task keeps its pipeline position and simply stops dispatching until the operator answers — a steer written to `context/<task_id>-STEER.md` plus an unblock (the next run for it receives the steer in its brief), a cancellation, or, for an initiative's children, anything that clears the propagated block (§6.1). A `blocked` report is an ordinary completion: the run exits, the lease frees, and the system moves on to other work while the question waits. Blocking consumes no review slot — the queue protects attention the system *pushes*; a blocked task waits for attention the operator must *choose* to give.

### Run / Lock / Runtime — see §10 (Dispatch) and §9 (Runtimes).

## 6. Task State Machine

Five statuses, and a status is the last stage a task **completed** — it advances only inside the transaction that completes the stage (Inv. 10). Nothing is ever "in progress" in the records; in-flight-ness lives in the lock, not the rows.

```
proposed ──promote──▶ seeded ──work/plan──▶ worked ──verify──▶ verified ──package──▶ packaged ──▶ operator decision
   │                   ▲  ▲                   │                                         │
   └─ reject           │  └─── correction ────┘                                         │
      (archived)       │       rally (≤ 3)                                              │
                       └────────────── refinement / operator revise ────────────────────┘
```

- **`proposed`** — born here, by Strategist(propose) or the operator. Leaves only through the Editor: promoted to `seeded`, or rejected (`decision = 'rejected'`, archived).
- **`seeded`** — owed execution: a Worker for a task, Strategist(initiative) for a drained initiative (§6.1).
- **`worked`** — owed verification.
- **`verified`** — owed packaging.
- **`packaged`** — in the review queue, out of pipeline dispatch. Not necessarily owed *nothing*: while its packet is unarchived and unsaturated it may re-enter through refinement (§8).

The same trigger enforces this machine for both scopes — an initiative's stages mean different things (§6.1) but move the same way.

- **Blocking is a flag, not a status.** `blocked = 1` (reason required) makes a row invisible to dispatch without destroying its pipeline position; unblocking resumes exactly where it stopped. Raised by a run that needs operator input, by the operator directly, or by child-block propagation (§6.1).
- **Terminal bookkeeping is `decision` + `archived`**, orthogonal to status: `approved` only from `packaged` (substrate-enforced), `rejected` at `proposed` by the Editor, `cancelled` by the operator at any stage. Archive requires a decision; archived rows are invisible to everything.
- **The expedite lane.** `priority = -1` partitions a row ahead of all stage ordering in dispatch (§10) — the tool for "drop everything and look at this."
- **Operator revise IS the refinement transition**: `packaged → seeded`, with the notes carried into the next run brief; the packet keeps its queue slot (Inv. 11). There are no superseding tasks and no version chains — the same row simply rides the pipeline again, rally and gates applying unchanged.
- **There is no dependency table.** Sequencing between units of work is Strategist(initiative)'s wave invariant (§6.1): only currently-actionable tasks are ever instantiated, so a dependency edge would only ever connect a row to a not-yet-row — and under the single global lease (Inv. 1), the parallelism a dependency graph would buy is nearly free to give up.

### 6.1 Initiatives and waves

The same machine carries an initiative; the stages change meaning, not movement:

| Status | For an initiative |
|---|---|
| `proposed` | a charter competing in the pool like any proposal |
| `seeded` | **active**. With open children: **parked** — owed nothing; the children are its dispatch presence. With none: **drained** — owed planning |
| `worked` | completion declared: Strategist(initiative) wrote a report mapping every charter criterion to evidence; owed arc verification |
| `verified` | owed the arc packet |
| `packaged` | arc packet in the review queue; approval writes the landing-pending mark and a following tick lands the branch (§7) |

**Waves.** Strategist(initiative) decomposes lazily: at each boundary it instantiates only the tasks that are actionable *right now*. Children are born `seeded` — their review is the Editor's **holistic plan review** of the wave, never a per-child contest in the proposal pool — and only into a live, still-seeded initiative (substrate-enforced).

**Strict drain.** The next wave (or the done-declaration) is owed only when *every* child is operator-decided and archived. The substrate refuses a done-declaration while any child is open.

**Maximally strict blocking.** One blocked child blocks the whole initiative — a trigger propagates it (`blocked_reason = 'blocked child #N'`) and no further planning happens until the operator weighs in. Unblocked siblings keep dispatching on their own flags. The propagated block auto-clears when the last blocked child is resolved or cancelled; a block the operator set directly on the initiative (any other reason) never auto-clears.

**The endgame reuses the pipeline verbatim.** Declaring done is the initiative's `seeded → worked`. The Verifier audits the criterion-to-evidence mapping *and* that the branch merges cleanly onto current main; failure is the ordinary correction rally back to `seeded`, where drain re-fires and Strategist(initiative) must answer the objections. The Packager renders the arc packet. Operator approval writes the landing-pending mark, and a following tick lands the branch onto main, archives, and deletes the worktree (§7); "not done yet" is the ordinary refinement round-trip `packaged → seeded` — drain holds, and the waves resume carrying the notes. Cancelling an initiative cascades: open children are cancelled and archived, their packets with them; main was never touched.

### 6.2 Git topology

This is contract enforced by `mc`'s runtime and the role briefs, not by SQLite. A standalone task gets its own ephemeral worktree off main, as always. Promoting an initiative cuts **one branch and one shared worktree** off main, and every child executes inside it — initiatives are assumed to carry breaking changes that must not land on main incrementally. A child's changes are committed, tagged with its task id, in the same breath as its Worker-stage completion: the Verifier verifies a commit, the packet cites a stable range, and a rare in-flight cancellation is one `git revert`. A killed or rejected child never had its changes committed at all. Strategist(initiative)'s first act at every wave boundary is merging main into the branch, so drift from interleaved Homie-filed tasks is absorbed in small doses and the final landing stays trivial. Abandoning an initiative never touches main.

**Mission Control's git runs in-container, at the canonical mount path.** Every mutating git operation Mission Control performs on a Worksource — worktree add/remove, branch, commit, merge, including the landing (§7) — runs only inside a container at the fixed canonical mount path; Mission Control never mutates a Worksource from the host. The operator's own git work is untouched: the workspace is their real repo, and the constraint binds Mission Control's objects only (`mc/task-<id>` branches, MC-created worktrees, landing merges) — never the operator's branches, worktrees, or merges. Because worktree metadata embeds a path to the parent repo's `.git` — and an absolute path written in-container does not resolve on the host, or vice versa — onboarding sets `worktree.useRelativePaths` on every Worksource repo, the container image ships git ≥ 2.48, and MC-created worktrees live in the same bind-mounted tree as the parent repo so the relative links resolve identically on both sides. Read-only git on the host is a requirement: the operator must be able to open a task's worktree in an editor and get working diff/log/blame. And no repo-wide git housekeeping ever runs in-container: operator worktrees living outside `workspace_root` are not mounted and look like missing directories from inside, so an in-container `git worktree prune` or `git gc` would destroy their metadata and break them on the host — Mission Control removes its own worktrees by exact path only.

## 7. Verification, Packaging & Review

### Promotion Gate Ladder
Before the Verifier passes work (`worked → verified`) it must clear every objective gate the task's criteria require (Inv. 12); judgments only the operator can make are *made decision-ready* in the packet — the packet *is* the human gate. The Verifier owns the ladder — staging deploy, the Playwright criterion walk, the security/dependency scan, recording each gate's evidence. Gates are **criterion-driven**: the ladder runs the gates the task's criteria bind, and a gate with no bindable criterion (no staging environment, no coverage tooling) is N/A, recorded as such in the evidence — the frozen Verifier directive says so, so Inv. 12 holds without inventing per-Worksource infrastructure.

- **coding:** lint/typecheck → tests pass → coverage Δ ≥ 0 → verifier approves → staging deploy + Playwright walk (screenshots/video) → security scan clean → Packager renders HTML with media inline.
- **research:** every claim → authoritative re-checked citation → cohesion pass → adversarial "what's missing/wrong" finds nothing material → recommended decision.
- **all:** each criterion binds a concrete evidence artifact; UI/UX changes bind a visual artifact; human criteria bind a decision prompt + evidence bundle.

### Decision outcomes
| Outcome | Conditions | Action |
|---|---|---|
| **PASS** | all objective gates clear, human judgments decision-ready | `worked → verified`; the Packager renders next |
| **CORRECT** | gates unmet, `correction_count < 3` | write the correction file; `worked → seeded`, `correction_count++` — the same row re-dispatches to a Worker with the feedback in its brief |
| **BUDGET SPENT** | `correction_count = 3` | `worked → verified` anyway; ships as an **exception-labeled** packet with unresolved gates in `risk`/`evidence`; recommended decision ≠ approve unless immaterial |

The bounded rally **is** the system's quality effort; the operator is the final reviewer via approve/revise/cancel. Reserve a genuine `blocked` report for real decision points, never to grade an unfinished output mid-rally.

### Operator decision flow
`mc packet decide <task> --approve|--revise|--cancel [--reason …]` (reason required for revise/cancel, forbidden for approve; asymmetric by design).
- **Approve is a pure state write**: `decision = 'approved'` plus, for a task whose record carries a branch, a durable **landing-pending** mark — no filesystem effect (operator verbs never dispatch, Inv. 2). The resident lands the work mechanically on a following tick (below); the review slot frees on landing success, never at approve time, and landing latency is at most one tick. A task with no branch — an artifact-plane deliverable (reports, analyses, rendered outputs under `artifact_roots`/`outputs/`) — has nothing staged against the workspace, so approve completes and archives it synchronously: no mark, no effect, no latency.
- **Revise** → the refinement transition `packaged → seeded`, notes carried into the next run brief; the packet keeps its slot (Inv. 11).
- **Cancel** → `decision = 'cancelled'`, archived (for an initiative: the cascade, §6.1); the worktree and branch are discarded, main untouched.

### Landing
Landing is deterministic execution of already-verified work — **agents are never in the landing path**. A landing-pending row makes the next tick's `mc dispatch` return a **land effect** (task, branch, verified SHA, target ref) ahead of queue selection (§10, step 0c); the resident effects it with a short-lived container running `mc-land` — a script baked into the base image and versioned with it — at the canonical mount path (§6.2):

1. **SHA fence.** Refuse unless the branch head equals the verified SHA, so only the exact reviewed commit can land.
2. **`git merge --no-ff <verified-sha>`, executed in the primary checkout** — never in a secondary worktree (git forbids checking out the target branch in two worktrees, and a merge that doesn't update the operator's working tree lands invisibly). Git's own dirty-tree protection is the second fence: uncommitted operator edits to files untouched by the merge survive it unchanged; uncommitted edits overlapping the merge make git refuse before writing anything.
3. On success, remove the task worktree and branch by exact path.

Exit 0 → the resident reports the landing through `mc`: the task archives, the packet with it, the slot frees, and the operator's working tree now contains the approved change. Nonzero (branch moved, dirty overlap, merge conflict) → `git merge --abort` where applicable, nothing half-lands, and the task blocks with the landing reason while the slot stays held; the operator commits/stashes, or a correction task reconciles, then a later tick retries the landing. Judgment re-enters only on failure, as ordinary pipeline work — correction, re-verify, land again.

## 8. Backpressure & Refinement

The Review Queue is the single global bounded buffer (Inv. 18): at most `REVIEW_WIP_CAP` (3) unarchived packets, enforced where it cannot be cheated — a substrate trigger aborts the birth of a fourth packet. Dispatch never *politely* respects the cap; it simply finds the queue full and changes mode (§10).

**At cap, the tick has exactly two moves:**
1. **Advance an in-flight refinement.** A packet whose task has re-entered the pipeline must be able to finish while the queue is full, or refinement deadlocks — its re-packaging is what eventually frees the slot. For an initiative's arc packet the in-flight work includes its wave children, so they keep dispatching at cap too.
2. **Start one.** Otherwise, take the best non-saturated packet: a **task** packet spawns the Refiner, which scopes a single deepening and re-enters the task (`packaged → seeded`); an **initiative** packet re-enters directly and Strategist(initiative) carries the deepening as ordinary waves. If every queued packet is saturated, **idle** — churning saturated work is negative value.

**Refinement is queue-neutral by construction.** The task round-trips `packaged → seeded → … → packaged` while its one packet keeps holding the slot (Inv. 11); re-packaging re-renders it in place. No version chains and no superseding tasks — the same row rides the pipeline again, the correction rally and the gate ladder apply unchanged, and producer ≠ judge (Inv. 9) holds with no special rule: a Verifier pass is a *genuine* deepening, a fail is *churn*.

**Saturation is computed, never hand-set.** Each refinement round-trip that fails to genuinely deepen the packet increments `refine_streak`; a substrate trigger marks the packet `saturated` at 3, dropping it from the refinement candidate set — it then waits only on the operator. A genuine deepening resets the streak.

**There is no separate quota-sink ladder.** Infra, tooling, and bench-research work are ordinary proposals: Strategist(propose) may pitch them, the Editor ranks them against everything else, and they flow through the same pipeline — when the queue has room. At cap the only spendable capacity is refinement, then idle.

Per pass (typed): research pulls/re-verifies sources + adversarial pass; coding rebases onto current main + re-runs the ladder + re-walks criteria; all sharpen thesis/recommended-decision.

## 9. Runtimes, Adapter & Routing

A **Runtime** is a native agent harness that executes a turn; Mission Control drives each subscription through its own harness and normalizes at the **Runtime Adapter** boundary (architecture A: native-harness-per-provider). All are subscription-billed by selection (Inv. 13).

**Two harness families today** (schema is provider-extensible):
- **`codex`** — `@openai/codex-sdk`/`codex exec`. ChatGPT-plan OAuth via the binding's canonical `.codex/auth.json` control dir, bind-mounted into the run (§11.4; denied to tools). Metered fallback (`CODEX_API_KEY`) bills **silently** — both harness/tool envs must omit it. Run signal = the structured stream (read only to detect completion and exit status for dispatch); native session file = `sessions/…/rollout-*.jsonl`. Session = `thread_id`. Self-orchestration mechanism: **native collaboration tool-calling** (§9.2).
- **`claude-sdk`** — the Claude Agent SDK, dispatched under one of two **model bindings** declared in `routing.md`: **`claude`** (Anthropic subscription, default endpoint) or **`minimax`** (MiniMax M3 via `ANTHROPIC_BASE_URL`/`ANTHROPIC_AUTH_TOKEN`, Subscription key). One harness, one adapter code path, two auth/endpoint configurations. Sharp edges (both bindings): `options.env` *replaces* the env (pass it fully); spawned subagents need `settingSources: []` / `--setting-sources "" --strict-mcp-config` or they fail with HTTP-200 malformed responses. Run signal = the `query()` stream (read only to detect completion and exit status for dispatch); native session file = `projects/…/<uuid>.jsonl`. Session = `session_id` with explicit lineage. Self-orchestration mechanism: **dynamic workflows** (§9.2).

**Runtime Adapter** is the single boundary that knows a harness's events, and it builds **no** trace of its own — Mission Control writes and maintains no bespoke trace or observability code (§13). It is the provider-specific layer beneath the in-container **agent runner** (§11.5): the runner owns the common lifecycle — launch the harness, drive top-level turns, forward shutdown, observe failure — while the adapter owns the Codex- or Claude-specific mechanics: how to start or resume its harness, submit a turn, recognize turn completion, and obtain the native session handle. The ownership chain is `host resident → container → agent runner → Runtime Adapter → agent harness/SDK → model`, and nothing above the adapter branches on provider event formats. The adapter reads the native stream only to detect run completion and exit status. Its other standing obligation is arranging that the harness writes its **native session file directly into the run's mounted session folder** (§4, §11.3) — Codex's `rollout-*.jsonl`, the Claude SDK's `<uuid>.jsonl` — so the record lands on durable host storage as it is written: no post-exit copy, nothing lost if the container dies mid-run. The spine keeps the four locators that find the record later — the folder's path, the harness that wrote it, the harness's own session handle, and the trace's filename (§15.4) — and every run's native file is **kept forever** (Inv. 26). The Homie tier additionally treats its native file as the resume substrate (§15.4); for pipeline runs it is purely the permanent post-mortem record, no longer a trace backstop because there is no trace to back up. Either way the durable native file is exactly what an operator who wants to *view* a run points their own optional Langfuse instance at (§13).

The capability work the adapter used to do *to build a normalized trace* — reconstructing Codex causality from `call_id` ordering, normalizing the two harnesses' usage shapes, folding self-orchestration into nested spans — is all retired with the trace layer. Self-orchestration plan capture survives only as an independent audit artifact (§9.2), not as trace input.

### 9.1 Role → Runtime Routing — mechanism is invariant, **values are tunable policy**
The **invariant** (Inv. 9): producer and judge of the same artifact run on decorrelated runtime families. The mapping itself lives in `routing.md` (the authoritative config; `mc doctor` and `mc dispatch` reject unresolved harness/model references, no fallback), retunable from the acceptance-rate signal (the cost/usage signal is gone with the trace layer — §13). The default policy places every **producing pipeline** role on the `claude-sdk` family and every **judging** role on the `codex` family, so Inv. 9's decorrelation holds pairwise by construction rather than by per-pair bookkeeping. Canonical binding ids are `chatgpt`, `claude`, and `minimax`; model strings are config values verified at onboarding's live no-op turn (§17.3), never compile-time constants:

| Role | Harness | Model binding | Why |
|---|---|---|---|
| Strategist | `claude-sdk` | `claude` | broadest generation for propose, initiative, and Console modes |
| Editor | `codex` | `chatgpt` | judge decorrelated from the Claude-family Strategist |
| Worker | `claude-sdk` | `minimax` | producer — subscription-subsidized depth; native dynamic workflows for breadth |
| Verifier | `codex` | `chatgpt` | judge decorrelated from the Worker's blind spots |
| Packager | `claude-sdk` | `minimax` | authors from the durable record; never judges |
| Refiner | `codex` | `chatgpt` | scopes deepenings; never produces them |
| Homie (§15) | `claude-sdk` | `claude` | the operator's conversational extension; sits in no producer/judge pair, so Inv. 9 constrains nothing here |

### 9.2 Self-Orchestration — fixed outer chain, adaptive inner flow

The outer topology never varies: Strategist(propose) → Editor → Worker → Verifier → Packager, plus Strategist(initiative), Strategist(console), and the Refiner, is frozen structure (§3, Inv. 20). Homie sits outside the leased pipeline as the operator's conversation role. What a role does *inside* its one leased run is not frozen: any role may orchestrate its own unit of work by fanning out read-only subagents (Inv. 14) — decomposing, parallelizing, and self-checking in whatever shape fits the task in front of it. Coordination becomes structure; judgment stays in the model. Each harness family provides this through its own mechanism, and the difference shapes what each is trusted with:

- **`claude-sdk` roles use dynamic workflows.** The role authors a small deterministic script (`export const meta` + `phase()`/`agent()`/`parallel()`, JSON schemas on agent returns) and the harness executes it: the orchestration moves into code *before* it runs, so its shape cannot drift mid-flight. **The trigger term is "dynamic workflow"** — directives and briefs use that exact phrase, which is what reliably activates the capability — and the `Workflow` tool is in every `claude-sdk` role's allowed tools. The generated script is captured verbatim from the Workflow tool input and persisted to `workflows/<run_id>-plan.js` (§4): the run's orchestration plan as a first-class, diffable audit artifact.
- **`codex` roles use native collaboration tool-calling** (`spawn_agent` → `wait` → `close_agent`). There is no deterministic script and no trigger phrase — coordination stays in the parent's context, step by step. That is acceptable precisely because of where `codex` sits in the routing: the judging roles want small, bounded fan-outs (a panel of skeptics, a per-criterion walk), and short shapes don't live long enough to drift. Codex directives therefore instruct the role to **select and execute a named orchestration pattern** — once it commits to a shape, the tool-calling follows — bounded rounds only, never an open-ended loop.

**Orchestrate-by-default.** Every role directive encodes orchestration as the *default posture*, not a granted permission: the role opens its run by planning the unit of work as an orchestration, and executing inline is the exception — taken only when the task is trivially single-context, and **stated**, one line of justification carried in the terminal `mc complete` report. The burden of proof sits on staying inline, never on fanning out. This is §1's economics applied one level down: the subscriptions make tokens the deliberately-spent input, and the failure mode this posture guards against is *breadth not covered*, not tokens spent — a few extra dollars of fan-out per Review Packet is the cheapest insurance operator attention can buy.

**Reference patterns.** Every directive names the same four role-agnostic patterns — a menu the role selects from, not a per-role prescription. Choosing from a vocabulary fires reliably where open-ended discretion does not; *which* pattern fits remains the role's judgment:

| Pattern | Shape | Natural fit |
|---|---|---|
| **Fanout-And-Synthesize** | independent slices run in parallel → one synthesizer folds the results | breadth scans, multi-surface research, evidence gathering |
| **Adversarial Verification** | N skeptic subagents try to *refute* the producer's claims, paired per claim-set | self-checking work before reporting `done`; criterion walks |
| **Generate-And-Filter** | many generators propose → a rubric + dedupe filter keeps the best | candidate generation, competing approaches, proposal shaping |
| **Tournament** | attempts judged pairwise; winners advance to a final | competing designs or drafts where exactly one must win |

The taxonomy's other two patterns are deliberately absent from the menu: **Classify-And-Act** is the dispatcher's own shape (the §10 control table *is* one, system-wide), and **Loop-Until-Done** belongs to the outer reconciler — an open-ended loop inside a leased run fights the lease clock, and is exactly the long-horizon shape that drifts.

**The escape valve.** Alongside the menu, every directive carries one general clause, phrased as Inv. 14's compressible-boundary test rather than "as you see fit": *beyond the reference patterns, author any orchestration whenever a sub-problem's full working context can be discarded once a small artifact (a brief, a verdict, a ranked list) returns; if you must hold the whole thing in your head to take the next step, do not spawn.* It is expected to fire rarely — rare-but-correct is what discretion is for; the menu and the default posture carry the routine load.

**What self-orchestration never changes.** Subagents are read-only, lease-free, and depth-1 (both harness families strip the spawn capability from children — runtime-enforced, not policed; Inv. 14). They run inside the parent's container and inherit its confinement wholesale: same mounts, same egress policy, same credential gateway, no new secret surface (§11). The dispatched role remains the sole writer of its unit of work. Intra-role adversarial verification is *self-checking effort*, never a substitute for the cross-runtime gates — the Verifier's verdict on a Worker's output (Inv. 9) is exactly as required as before, however many skeptics the Worker ran on itself first. A subagent's work is visible only in its parent's native session file (§9) — there is no separate trace layer that records it.

## 10. Dispatch Architecture

Mission Control is a **durable-state reconciler**, not a message bus. Agents are ephemeral, so dispatch is a deterministic function of durable state, evaluated by `mc` each tick.

**Two kinds of state:** **Records** (`tasks` + `review_packets` — canonical truth) and the **Lock** (the single in-flight slot + lease). There is no third kind — no activations table, no dispatch index, no stored queue. Every tick recomputes "who is owed what" directly from the records: `status` is the last stage completed (Inv. 10), so the rows *are* the queue, and there is nothing to drift, rebuild, or reconcile against them.

### Execution Lock (lease)
Singleton row in the spine, CAS-only — structured state, never a `lock.json` on the file plane. Fields: `run_id`, `worksource`, `subject`, `owner` (`strategist`|`editor`|`worker`|`verifier`|`packager`|`refiner` — the leased pipeline role holding the lease, flat and untyped; Strategist mode lives in the run brief), `acquired_at`, `last_heartbeat_at`, `hard_deadline_at`, `timeout_minutes` (60), `grace_minutes` (15), `heartbeat_interval_s` (60), `spawn_grace_s` (60). `subject` is **nullable**: a propose-style run — Strategist(propose) or Strategist(console) — holds the lease with no subject task and no synthetic placeholder row.

The row carries **two distinct time bounds** (Inv. 1): `last_heartbeat_at` is the renewable last-seen time that detects a runner that disappeared; `hard_deadline_at` is a non-renewable hard deadline for the run. A heartbeat may advance the former but can never extend the latter, so a hung-but-live run is bounded too. The `run_id` in the row is also the fencing token that prevents a late completion from an old container being applied to a newer run.

**The lock carries no Record status.** `owner` names *who holds the lease right now*, never *what state the subject is in* — that lives only in the Records. The lock supplies the two things a status field structurally cannot: **liveness** (the heartbeat the reaper acts on, so a crashed run's silence doesn't read as busy forever) and **identity** (which run to fence on completion). The reaper acts on the lease without reasoning about the state machine at all; the Records carry the subject the rest of the way.

**The heartbeat is the runner's, not the model's.** The pipeline agent runner (§11.5), not the agent, emits the run's periodic heartbeat through a private, lifecycle-only `mc` scope: it begins only after the adapter has established the harness session, continues while the runner is alive and the harness has not exited, and stops when either one terminates. It proves only that the trusted execution controller and its harness process still exist — never that the model is making semantic progress; progress is bounded by the hard deadline. The agent's role-scoped `mc` capability cannot heartbeat or extend its own lease (the enforcement grain of that scope split — per-container kernel-backed, within-container best-effort — is §11.5's).

**Crash detection** (the conditions the tick evaluates):
- **First-heartbeat watchdog (`spawn_grace_s`).** A *stamped-but-never-heartbeated* slot is a failed spawn (the post-commit gap, or a cold-start that never came up); reaped in seconds, not after ~75 min.
- **Lease timeout (`timeout + grace`).** A run that heartbeated then went silent is reaped on the full lease.
- **Hard deadline.** A run still heartbeating past `hard_deadline_at` is reaped regardless — liveness is not progress.

**Two budgets, distinct:** `dispatch_retries` on the task (infra/dispatch death — decremented only by the reaper) ≠ `correction_count` (rally quality, ≤ 3). A reaped run never charges the rally budget, and a quality miss never charges the infra budget. There is no autonomous semantic-retry loop: a Worker that cannot do the work reports `blocked` with the reason, and the operator decides.

### Control flow — one tick, walked top-down

**Where dispatch runs, and who effects it.** `mc dispatch` is invoked from exactly one place: the resident's tick loop, a plain in-process interval timer firing every `tick_interval_s` (60s default). One tick runs at a time — a firing that lands while a tick is in flight is skipped, never queued or run concurrently — and the resident fires an immediate tick on wake-from-sleep (macOS pauses timers during sleep). The resident is the sole holder of spawn/effect authority; an event-nudged tick (container-exit or inbound-commit triggers feeding the same single loop) is a possible future iteration, explicitly out of scope for v1. `mc dispatch` executes inside the lock domain (in the warm helper when host-originated, §11.5): it reconciles the records, picks the next (role, task), claims the lease, and opens the run, allocating its id and therefore its session-folder path — pure record-keeping inside the database. When that committed decision requires host authority to realize — creating the session folder, materializing `run.json`, wiring the proxy, starting or stopping a container — the `mc` invocation returns **structured effect data** to the resident as its ordinary command result: not a second source of truth and not a durable handoff file, just the instruction sheet for a decision already committed in the spine. The resident effects it in order — folder, `run.json`, proxy, container — so the mount target always exists before the mount (§11.3). Agents are never in the spawn path (Inv. 2).

Dispatch runs only when the lease is free; each tick commits exactly one action (Inv. 3).

**(0) Reconcile the lease.** Fresh → return. Reapable — heartbeat stale per the thresholds above, or hard deadline passed → reap: mark the run reaped, decrement the subject's `dispatch_retries`, free the lease, and return the container stop to the resident as effect data (only the host side holds the control socket, §11.6). There is nothing else to undo — the crashed run never advanced its task (Inv. 10), so the queries below simply re-select it. At `dispatch_retries = 0`, set `blocked` with the reason instead (never a silent loop). Reaping a **subjectless** run (Strategist(propose)/console, `subject = NULL`) marks it reaped, frees the lock, and emits the stop effect while charging nothing — there is no subject to decrement; the same dispatch branch simply re-fires on a later tick.

**(0b) Console due?** If the Daily Console's stored delivery time has passed with no same-day `daily.briefing` event, dispatch Strategist(console) (§14) and stop — the one time-triggered dispatch in the system, riding Inv. 21's stored-schedule carve-out.

**(0c) Landing pending?** A landing-pending row (§7) makes this tick return exactly one **land effect** — task, branch, verified SHA, target ref — ahead of queue selection, and stop; with several rows pending (the operator approved more than one packet between ticks), one lands per tick in a fixed deterministic order until drained. Landing-pending rows are invisible to everything below: never selected by the with-room query (they sit at `packaged`, rank 0) and never selected for refinement by (2b) — a decided packet is out of the refinement pool, or dispatch would spawn a Refiner on already-approved work — while the packet still counts toward queue occupancy, because the slot frees on landing success, not at approve time. A reported landing success archives the task, cascades to the packet, and frees the slot; a reported failure blocks the task with the landing reason, archives nothing, and keeps the slot held (§7).

**(1) Queue occupancy.**
```sql
SELECT COUNT(*) FROM review_packets WHERE archived = 0;
```

**Queue at cap (= 3):**

**(2a) Advance an in-flight refinement first.** A packet whose task has re-entered the pipeline must be able to finish while the queue is full, or refinement deadlocks (§8).
```sql
SELECT t.id, t.status, t.scope FROM tasks t
JOIN review_packets p ON (p.task_id = t.id OR p.task_id = t.initiative_id)
WHERE p.archived = 0 AND t.archived = 0 AND t.blocked = 0
  AND t.status <> 'packaged'
  AND (t.scope = 'task' OR NOT EXISTS
       (SELECT 1 FROM tasks c WHERE c.initiative_id = t.id AND c.archived = 0))
ORDER BY t.stage_rank DESC, t.priority, t.created_at
LIMIT 1;
```
The join's two arms matter: `p.task_id = t.id` matches a packet's own task that has round-tripped back into the pipeline; `p.task_id = t.initiative_id` matches the wave children of a packet-holding initiative — without it, an initiative bounced back from operator review could never advance its refinement waves at cap. The selected row maps to a role by the same (status, scope) table as (3).

**(2b) None in flight → start a refinement on the best non-saturated packet.**
```sql
SELECT t.id, t.scope FROM review_packets p JOIN tasks t ON t.id = p.task_id
WHERE p.archived = 0 AND p.saturated = 0 AND t.decision IS NULL
ORDER BY t.priority, p.created_at
LIMIT 1;
```
Scope `task` → spawn the **Refiner**. Scope `initiative` → re-enter `packaged → seeded`; Strategist(initiative) carries the refinement as ordinary waves, which (2a) then picks up. Empty result ⇒ all packets saturated ⇒ **idle**.

**Queue has room (< 3):**

**(3) The single next dispatch.**
```sql
SELECT id, status, scope FROM tasks
WHERE archived = 0 AND blocked = 0 AND stage_rank > 0
  AND (scope = 'task' OR NOT EXISTS
       (SELECT 1 FROM tasks c WHERE c.initiative_id = tasks.id AND c.archived = 0))
ORDER BY (priority <= -1) DESC, stage_rank DESC, priority, created_at, id
LIMIT 1;
```
The ordering reads left to right. The `(priority <= -1)` prefix is the **expedite lane**: a P-1 row partitions ahead of everything, so a fresh urgent proposal pulls the Editor ahead of the backlog instead of starving beneath it. Within a partition, **furthest-first** (`stage_rank DESC`): finish what is closest to done before starting anything new. Then priority, then age. An initiative with open children is **parked** — the `NOT EXISTS` arm hides it; its children are its dispatch presence — and a blocked initiative is excluded while its unblocked children keep flowing on their own flags.

| (status, scope) | Dispatch |
|---|---|
| `proposed`, any | **Editor** — the one **batch-wise** stage: the brief snapshots the *entire* proposed pool for contrastive rank-then-cut, scopes mixed, never this row alone |
| `seeded`, task | **Worker** |
| `seeded`, initiative | **Strategist(initiative)** (drained ⇒ owed the next wave, or the done-declaration) |
| `worked`, task | **Verifier** |
| `worked`, initiative | **Verifier** (arc: completion report + clean merge onto current main) |
| `verified`, any | **Packager** (task packet / arc packet) |

**(4) Empty result ⇒ nothing dispatchable ⇒ spawn Strategist(propose) to seed.** Covers "no tasks at all", "only blocked or packaged rows", and "only parked initiatives" — Strategist(initiative) needs no idle hook, because a drained initiative surfaces through (3) ahead of this fallback. Strategist(propose)'s brief carries the dedupe memory:
```sql
SELECT title FROM tasks WHERE decision = 'rejected'
ORDER BY decided_at DESC LIMIT 20;
```

### Completion is a pure write
The completing agent itself invokes `mc complete` through its role-scoped capability — there is no report file and no replay, and harness exit alone never advances pipeline state: a run that exits without an accepted terminal action is recovered by the lease rules above. `mc complete` writes the role's report, advances the subject's status in the same transaction (Inv. 10), releases the lease, and returns — **it never dispatches**. The next timer tick selects the follow-on stage from the written records, so the pipeline flows tick-to-tick with at most `tick_interval_s` of stage-transition latency (Inv. 3). No `mc` verb dispatches: completions, operator actions, and conversation traffic all only write state; the resident's tick is the one dispatch call site (Inv. 2, 3).

### Crash recovery is mostly the absence of a problem
Because no run mutates its subject before completing, recovery has no rollback step: reap, decrement `dispatch_retries`, re-select. The role-specific notes that remain:

| Reaped owner/mode | What the next tick sees |
|---|---|
| `worker` | task still `seeded`; re-selected by the same query. An initiative child's partial work is uncommitted in the shared worktree — inert (commit happens only at completion, Inv. 25) — so respawn-and-continue and host-side `git restore` are both safe |
| `verifier` / `packager` | task still `worked` / `verified`; re-selected; `correction_count` untouched |
| `editor` | pool intact; the batch re-fires over the whole pool |
| `strategist(initiative)` | initiative still drained-and-seeded; the boundary re-fires (a wave insert is one transaction — whole wave or nothing) |
| `strategist(propose)` / `strategist(console)` / `refiner` | nothing inserted / delivered / re-entered; the same branch re-fires |

### Concurrency fencing
Claim is CAS on the lock row inside the committed `BEGIN IMMEDIATE` transaction; a losing racer sees a non-empty lock and aborts before any effect data is returned — so the resident is never handed the same spawn twice. A process-level `mc.dispatch` flock, held on the spine volume (in the lock domain), additionally prevents wasted concurrent evaluation. `mc complete --run <run_id>` is fenced: a completion whose run no longer matches the lease is rejected, never double-applied — and `mc heartbeat` is fenced the same way, so a zombie run can neither renew its old lease nor touch the new holder's.

**Clock and storage discipline.** All lease math uses one clock — the lock domain's own: every timestamp is written by `mc` with `datetime('now')` inside the transaction, and host processes never pass timestamps in. The spine runs WAL mode with `busy_timeout` set and every mutation under `BEGIN IMMEDIATE`.

## 11. Sandboxing & Containerization

§5's Sandbox Profile says what a run may touch; this section says how the container enforces it. **Every run is containerized — there is no host-execution mode.** The machinery is transplanted from NanoClaw's container layer — bind-mount mount plans, a credential-injecting gateway, label-scoped cleanup — with one deliberate change of grain and several fail-open defaults inverted to fail-closed.

The change of grain: NanoClaw confines **per agent group**, because there the group is the unit of jurisdiction. Mission Control's unit of jurisdiction is the **Worksource** (Inv. 22), so confinement follows it. No new invariants are introduced here; containerization exists to make invariants 16 (auth not tool-visible) and 22 (scoped jurisdiction) *structural*: true because the state that would violate them does not exist inside the boundary at all.

### 11.1 Unit of confinement: per-Worksource identity, per-run instance

Containerization has two grains, and it matters which thing lives at which one.

- **The container identity is per-Worksource.** Image, mount plan, network policy, synthetic home, and secret-binding scope are all properties of the Worksource's Sandbox Profile. Every role that runs against a single Worksource — Strategist, Editor, Worker, Verifier, Packager, and Refiner — runs inside the *same* confinement, because they all hold the same jurisdiction. The one exception is a **cross-Worksource Strategist(propose) seeding pass**, which runs under the read-only **seeding-scope profile** (Inv. 22): it may mount several Worksources' workspaces read-only to compare leverage and writes no deliverable, so the shared-confinement rule that protects producers from one another does not apply to it. It is otherwise an ordinary pipeline dispatch — control step (4) fires it only when nothing else is actionable, it claims the global lease like any role run (holding it freezes nothing, since the row's own trigger condition is that no other work exists), and its read-only subagent scouts are lease-free exactly as Inv. 14 makes every subagent. Only its *mount plan* is special. Separating producer from judge is a session-and-runtime concern (Inv. 9), not a filesystem one: the Worker and the Verifier see the same mounts, and what keeps them honest is fresh sessions on decorrelated runtimes, not different walls.
- **The container instance is per-run.** Every dispatch starts a fresh container (`--rm`); it exits when the role self-terminates (Inv. 5). Nothing accumulates in container-local state between runs — anything durable lives on a mounted path or it does not survive. Because the global lease admits one run system-wide (Inv. 1), at most one **leased pipeline** container exists at any moment; there is no container pool and no warm reuse. The one lease-free tier — a Homie operator session (§15) — may legitimately run alongside it, which is why containers carry a tier label and the lease is never the liveness test for Homie sessions (§11.6). The cross-Worksource seeding pass is *not* a separate tier: it is a leased pipeline run that merely carries a different mount plan.

This is the main departure from NanoClaw, whose containers are semi-resident (woken per session, polling for messages, idle-reaped). Mission Control's roles are ephemeral by invariant, so its containers are too: the dispatch lifecycle *is* the container lifecycle. A resident warm *agent* container was considered and rejected — it would save only seconds of startup in a system whose cadence is minutes, and it would re-import the entire stuck-container supervision subsystem (idle ceilings, claim-age checks, wake deduplication) that NanoClaw needs precisely *because* its containers are resident. (The warm helper of §11.5 is not a counterexample: it runs no agent and no harness, existing only as a warm place for host-originated `mc` invocations. Nor is the Homie container, whose within-conversation warmth is a latency optimization at the lease-free tier, never a correctness dependency — §15.5.)

Each instance is named `mc-<worksource>-<run_id>` and stamped with a static `mc-managed` label, an **`mc-tier` label (`pipeline`|`homie`)**, plus the Worksource and run ids — the handles for orphan cleanup (§11.6) and observability. The tier label exists because the two tiers have different liveness records: pipeline runs hold the lease, Homie sessions hold a registered conversation. (NanoClaw additionally stamps a per-install slug so two installs sharing one Docker daemon cannot reap each other's containers; Mission Control is deployed once per machine, so the slug is deliberately dropped.)

### 11.2 Images

One **base image**, shared by all Worksources, containing what every run needs: the privileged setuid `mc` binary (baked in, never bind-mounted — the setuid bit must live on the image filesystem, §11.5), the two runtime-harness CLIs at pinned versions (pinning is deliberate — an unpinned install means every rebuild silently picks up a new harness and can break every Worksource in lockstep; the pins live as exact-version Dockerfile `ARG`s, in artifacts rather than prose), Node and Bun, git ≥ 2.48 (relative worktree links, §6.2), a browser for the Verifier's Playwright criterion walk, and `tini` as PID 1 so signals are forwarded correctly and the runner shuts down cleanly on SIGTERM instead of being orphaned. The image is built once at onboarding and pinned by digest, with the Playwright browser layer kept separate so most rebuilds skip it.

The in-container runner's source code (§11.5) is **bind-mounted read-only, never baked into the image** — a source change never requires an image rebuild.

**There is no per-Worksource derived image in v1.** Software a run needs falls into three layers, and only the third would ever justify image machinery:

1. **Project dependencies** (npm/pip/cargo manifests in the repo) live in the workspace mount, are installed by the Worker as part of the work, and land in the commit. The image is never involved.
2. **Ephemeral user-space tools** (`npx`, `uvx`, `pipx`, `cargo install` into the synthetic home) are installed on demand, need no root, die with the container, and are constrained only by the egress policy like everything else the agent does. To keep on-demand cheap across per-run containers, each Worksource gets a persistent **package-cache mount** (pnpm store, uv/pip cache) in its per-Worksource state dir.
3. **System-level packages** (apt: `ffmpeg`, `libpq`, an extra language runtime) are the one layer on-demand handles badly — root in the agent's hands, re-installation on every per-run container, environment drift between producer and judge. This layer is **deliberately deferred**: the upgrade path is a known, cheap shape (a one-layer `FROM base` derived image per Worksource, NanoClaw's `buildAgentGroupImage`, ~40 lines), added the first time a Worksource genuinely hits an apt-level need.

### 11.3 The mount plan

A container starts from nothing; what a run can see is exactly the enumerated mounts. This flips `denied_paths` from a fence the host must police into a **validation predicate**: at profile save and again at every spawn, no resolved mount may fall under another Worksource's roots or under a host credential directory. Deny-by-default does the rest.

| Container path | Host source | Access | Purpose |
|---|---|---|---|
| `/workspace` | container-local scratch | RW | staged outputs, evidence captures — dies with the container; anything durable leaves through the artifact roots or `mc` |
| `/workspace/source` | profile `workspace_root` | RW | the Worksource's actual workspace (repo, vault, …) |
| `/workspace/artifacts` | profile `artifact_roots` | RW | durable outputs, rendered packets |
| `run.json` | materialized by the resident at spawn (§10) | RO (nested, ephemeral) | the run's trusted launch envelope: run/conversation id, tier, harness and model binding, fresh-or-resume mode, mounted paths, and the prepared opening brief (§11.5); mounted separately from the session folder and removed with the container |
| the harness's session-write path | per-run session folder `sessions/<run_id>/` | RW | **trace-only** (Inv. 26): the harness's native session file lands here durably as it is written. Each container mounts **only its own** folder — one run's verbatim record is invisible to every other run. A Homie resume **re-mounts this same folder** (§15.4); no restore-copy |
| profile `readonly_mounts` | operator-listed paths | RO | reference material the work needs but must not change |
| `/home/agent` | per-Worksource synthetic home | RW | harness settings and non-trace session state; never the operator's real HOME |
| `/app/src` | shared runner source | RO | the in-container agent runner (§11.5), identical for every Worksource |
| the spine volume | runtime-local named volume (Inv. 24) | RW to `mc` only | the database; files owned by the privileged uid, unreadable to the agent uid — reachable only through setuid `mc` (§11.5) |

Two NanoClaw techniques carry over directly. First, the **nested read-only mount**: `run.json` and the sandbox config are mounted RO *on top of* an RW directory, so the agent can read its own configuration but cannot rewrite its own confinement. Second, **materialize-at-spawn**: `run.json` is regenerated from the dispatch decision on every launch and removed with the container — the database is the source of truth and the file is only the conveyance, so config drift between DB and disk is structurally impossible.

**Mount validation is fail-closed.** Operator-added mounts are checked against an allowlist file that lives in the operator's config directory, *outside every mounted root*, so no agent can edit the rules that confine it. Validation resolves symlinks to real paths before comparing; rejects any path matching a blocked-pattern list (`.ssh`, `.aws`, `.kube`, `.gnupg`, `credentials`, `.env`, private-key names, …); requires container paths to be relative, free of `..`, and free of `:` (a colon would smuggle extra options into the runtime's `-v` argument); and grants read-write only when the mount requests it *and* the allowlist root permits it. The inversion from NanoClaw: where NanoClaw logs a rejected mount, drops it, and spawns anyway — so the run proceeds with silently different semantics — `mc` **aborts the dispatch and blocks the subject task with the reason**, surfacing it to the operator. A run whose confinement plan could not be fully applied is exactly the run that must not start.

### 11.4 Credentials: the egress gateway

The strongest property container mode buys is that **subscription credentials never enter the container at all**. The resident, effecting the spawn (§10), wires each container to a run-scoped **credential-injecting egress proxy** it hosts (Inv. 23): the container receives only `HTTPS_PROXY` and a trusted CA certificate, and the proxy — which alone holds the runtime auth material — injects the authentication on provider API calls as they pass through. Inside the boundary there is no `auth.json`, no token in any env var, nothing for a model-invoked tool to read or exfiltrate. Inv. 16 stops being a discipline and becomes a fact about what exists — for the bindings whose credentials can travel this way; which ones can is decided per binding at the end of this section.

This inherits NanoClaw's posture toward its gateway, which is the correct one: **if the gateway cannot be provably wired, the container does not spawn.** This is the same fail-closed posture as mount validation extended to credential delivery — a run whose credentials could not be delivered through the gateway as planned, or that would fall back to ambient auth, never starts.

`network_allow` is validated the same way, and for the same reason. Its rules are checked when the profile is saved and again at spawn, exactly as mount rules (§11.3) and the gateway are; a rule that cannot be applied — a name that will not resolve, a port the network layer refuses — aborts the dispatch and blocks the subject task with the reason rather than starting a run with a different network shape than the profile declared. A run whose network plan could not be fully applied is, like one whose mounts or credentials could not, exactly the run that must not start.

The same proxy is the enforcement point for the profile's `egress_policy`, and the network layer blocks all non-proxy egress except the profile's `network_allow` host:port rules, so the proxy cannot be bypassed for anything not explicitly enumerated:

| `egress_policy` | Effect at the gateway |
|---|---|
| `none` | only the runtime providers' API endpoints pass; all other egress refused |
| `allowlist` | provider endpoints plus the profile's listed domains |
| `open+audit` | everything passes; every contacted host is written to the run's egress audit log (the proxy's own record, not an agent-session trace) |

**Egress has two planes, and they are independent.** The table above governs only the first one. HTTP egress is enforced by the proxy and addressed by domain — that is what `egress_policy` configures. Raw-TCP egress is enforced by the network layer and addressed by host:port — that is what `network_allow` configures. Neither one ever implies the other: opening a domain for HTTPS does nothing for a Postgres socket, and opening a port for raw TCP does nothing for the proxy. Three rulings follow from this split, and they are design decisions, not phrasing:

- **`open+audit` opens only the HTTP plane.** Even under `open+audit`, raw TCP is never blanket-open; it is always the explicit enumerated `network_allow` list. The reason is the audit property that makes `open+audit` tolerable in the first place: the proxy sits in the path and logs every host it passes, so "open" still means "observed." Raw TCP has nothing terminating it to do that logging, so a blanket-open raw-TCP mode would be an un-auditable channel. Keeping raw TCP allowlist-only at all times is what preserves the meaning of "audit" — and it is why the right change is a separate field rather than a fourth `egress_policy` mode.
- **A `network_allow` entry pairs with the credential's provisioning.** Opening port 5432 is only half of reaching a database; the other half is the connection string, which the operator provisions in the workspace themselves (§5). The two move together — open the port and provision the credential — because a network hole with no secret, or a secret with no hole, is a half-configured run.
- **Raw-TCP egress is auditable at rule granularity only, not per-connection.** This one is an honest limitation rather than a property to celebrate. You can see that the profile permits `…:5432`; you cannot get a per-call host log the way `open+audit` gives you for HTTP, because nothing sits in the path to record each connection. In the egress audit log and in System Health (§13), a `network_allow` entry therefore appears as a **declared standing allowance**, not as a stream of audited events. That is the inherent cost of bypassing the proxy, named here so it is not mistaken for an audit gap.

**Worksource tool secrets are different from runtime auth** and are not Mission Control records (§5): a deploy key or MCP credential is an operator-managed file in the workspace, read by tools like any other workspace file and scoped by the mount plan. The split is: runtime auth never inside the container as agent-readable material; tool secrets inside only as workspace content the operator put there.

**Delivery is tiered per binding, decided.** Header injection assumes the credential is a static string the harness sends blindly. That is true of the `minimax` binding (a bearer key in a header) and false of the two OAuth bindings: `codex` (ChatGPT plan) and `claude` (Anthropic subscription) hold a self-refreshing access/refresh token pair that the harness reads from its own control files, validates before starting, and rotates on refresh. Hand such a harness a dummy and it does not send a swappable header — it decides it is not logged in, or falls back to metered API-key auth, which Inv. 13 forbids (observed directly: proxying codex forced exactly that fallback). And even a successful header swap dies at the first expiry, when the harness tries to refresh with the dummy it holds. The only full-gateway repair is a proxy that owns the OAuth lifecycle itself — intercepting refresh attempts and impersonating an undocumented login flow that breaks whenever the harness's auth changes, and whose refresh-token rotation can invalidate the operator's real login. That cost is declined.

So `runtime_auth_delivery` is `gateway` for static-token bindings, and `materialized` is the shipped posture for the OAuth bindings — where materialization means **bind-mounting canonical state, never copying**. Each OAuth binding has one canonical `runtime_control_dir`, host-side and owned by the resident; a run that uses the binding gets that dir bind-mounted read-write into its container, with no copy-in step. When the harness self-refreshes mid-run, the rotated tokens land in the canonical files and every subsequent run starts from current tokens — necessary because both OAuth harnesses self-refresh and refresh-token rotation can be single-use, so a refresh inside a container-local *copy* either dies with the container (canonical copy stale) or invalidates the canonical copy server-side (every later run broken). Serialization is inherited, not built: Inv. 1 guarantees at most one pipeline run at a time, so pipeline runs never share the dir concurrently — no lock, no copy-back protocol, no reconciliation logic. One declared caveat: a Homie session and a pipeline run using the *same* OAuth binding can be live concurrently and share the dir; the simultaneous-refresh race there is accepted for v1 and must be stated in the operator-facing docs. `mc doctor` includes an **auth-health probe per OAuth binding** (a no-op turn or token-validity check), so an invalidated token surfaces as a diagnostic finding instead of a failing run. Materialization remains a **declared downgrade, never a silent fallback**: it is stamped on the profile and recorded in the run's audit record, and its damage is bounded by the rest of the boundary — the egress allowlist confines where a token can be sent from inside, and subscription tokens carry no metered spend. Inv. 16 is therefore a *fact* for static-token bindings and holds only as far as the harness's own discipline for OAuth ones; that residue is worth revisiting if a provider ever offers scoped or short-lived run tokens, not worth a proxy-owned OAuth client today. Note the proxy still fronts a materialized run's egress — materialization changes who holds the credential, not the enforcement of `egress_policy`.

### 11.5 The boundary: privileged `mc` and the agent runner

**The spine never leaves the lock domain — and agents cannot open it even from inside.** Agents run inside the lock domain, so they *could* in principle open the spine file directly and bypass the binary. The architecture forecloses that at the kernel level: the agent's own OS identity is denied access to the spine, and the privileged `mc` binary is the single broker permitted to open it. An agent that tries to read or write the database any other way is refused by the kernel, not merely discouraged by convention. This makes `mc` a **real gate rather than an advisory cooperator** (Inv. 2), with two consequences the system depends on:

- **Sole-writer integrity holds as a guarantee.** Only `mc` can write, because only `mc` can open the database for writing; concurrency is serialized by the lease and the single lock domain.
- **Read-scope holds as a guarantee** (Inv. 22). An agent cannot read the raw database at all, so it cannot see beyond its own jurisdiction — `mc` performs reads on its behalf and applies the scope. This is strictly stronger than a mounted-but-trusted model, where any run could read every other Worksource's tasks, packets, and blocked-task contents.

The gate's strength equals the strength of the surrounding container isolation: it assumes an agent cannot escalate privilege inside its own container — the same trust boundary already relied upon to keep agents from escaping to the host, so no new assumption.

**The mechanism is ownership plus setuid, decided.** `mc` is a single statically-linked binary (Go — the kernel ignores the setuid bit on interpreted scripts, and a static binary leaves no interpreter or env-injected code path to sanitize), baked into the base image at a pinned version with its setuid bit set, owned by a dedicated privileged uid. The spine's named volume is mounted into every container, but every file in it is owned by that same privileged uid with no access granted to the agent's uid — the kernel refuses any direct open, and executing `mc`, which escalates to the owning uid, is the only road to the bytes. Two hardening preconditions become requirements rather than suggestions: agent containers must not run with `no-new-privileges` (it disables setuid outright), and the binary must sit on a filesystem that honors the suid bit — which is why it is baked into the image while the runner's source remains a read-only bind mount (§11.2).

A direct payoff: the Homie ingests external input and is the prompt-injection target, but it is gated identically — an injected agent cannot bypass `mc` to read or corrupt the spine; the worst it can do is misuse the scoped verbs it was already permitted to call, which collapses the injection threat from "touch the whole spine" to the ordinary authorization problem.

**Host components use the real CLI too.** The resident and dashboard are native, so on macOS they cannot open the spine directly (Inv. 24). They reach it the same way agents do — by invoking the real `mc` binary — except across the host→runtime boundary, into a **warm helper** that sits inside the lock domain: a long-lived container that runs no logic of its own, existing only so the frequent host-side spine access the chat loop requires pays no cold start. On a Linux host there is no separate kernel, no helper, and no boundary crossing (§12). The command is the real CLI and the host process reads its output; the boundary is only ever crossed by real `mc` commands and their results — never by a file or an RPC. There is no stub, no invented protocol, and no durable handoff file.

**The crossing mechanic is self-delegation, decided.** `mc` is one Go binary cross-compiled for both platforms; at startup it determines whether it can reach the spine directly (inside the lock domain) and, when it cannot — every darwin invocation, whether the resident's or the operator's shell — it re-invokes itself via the container runtime's exec API into the labeled helper, passing argv, stdin/stdout/stderr, and the exit code through untouched. The host-installed `mc` and the image's `mc` are the same artifact at the same pinned version, so nothing drifts; the transport is the runtime's own exec channel carrying a real `mc` command line, so the no-protocol rule above is satisfied by construction. The alternatives were rejected for concrete reasons: a host wrapper script is a second versioned artifact with shell-quoting failure modes, and routing operator invocations through the resident makes the resident a dependency of `mc doctor` — the tool needed precisely when the resident is broken.

**The helper's lifecycle follows from its statelessness.** It is created by onboarding (§17), ensured running by the resident at boot and each tick, and lazily recreated by any host `mc` invocation that finds it missing — failing closed if the runtime cannot start it. It carries the mc-tier labeling like every other container, mounts the spine volume, and holds no state of its own, so it can be killed and recreated at any moment with no drain or handoff; an image upgrade replaces it rather than updating it. It mounts no `run.json`, so `mc` executed inside it carries no run identity and naturally lands in the operator/host verb scope — scope follows from what is mounted, in both directions. It is not the egress proxy, not a supervisor, and not a broker; its only failure mode is "not running," and it adds no security surface beyond the runtime's exec API, which agent containers never receive.

**The agent runner.** Every agent container starts one trusted Mission Control **agent runner** as its entrypoint (source bind-mounted RO, §11.2); the runner launches the configured harness through the Runtime Adapter (§9) and drives its top-level turns. It has no agent intelligence and makes no semantic decisions — a mechanical control loop around the harness. Its trusted startup input is the ephemeral, read-only **`run.json`** launch envelope, prepared by the resident from the dispatch decision before launch (§10): the run or conversation id, tier, selected harness and model binding, fresh-or-resume mode, mounted paths, and the already-constructed top-level agent input — for a pipeline run, the complete opening brief (role, task, criteria, steer/correction context, record and artifact references, and the fencing `run_id`). It contains no runtime credentials, and mechanically reading it means validating a fixed schema, never interpreting a task. For both tiers, the runner: starts or resumes the harness through its adapter; supplies one top-level turn at a time and waits for its result; installs the tier's tool and `mc` capability scopes before the model starts, keeping runner-only capabilities out of the model's tool environment and out of subagents; forwards shutdown signals, observes harness failure, and exits with the container; and ensures the harness writes its native session record into the mounted session folder as it runs. It does **not**: choose a task, role, Worksource, harness, or model; acquire work; decide whether work passes or which transition should occur; create folders, worktrees, containers, or successors; interpret natural language into Mission Control actions; open the spine directly; hold surface connections or credentials; or decide whether, when, or how the top-level agent self-orchestrates (§9.2 — that is harness and agent behavior).

**Pipeline runner — one run, one top-level turn.** It starts a fresh harness session, passes the preconstructed brief unchanged, drives exactly one turn, and exits when the harness terminates; it does not poll for work — dispatch already selected it and the resident already launched the container. It is not an `mc` client in the role's semantic path: the pipeline agent holds the real, role-scoped `mc` capability and invokes it directly for every intentional read or mutation, including its terminal completion or blocked report. The runner never proxies, replays, or infers an `mc` invocation from the model's final text, and never converts an ordinary harness exit into success — a run that exits without an accepted terminal action is recovered by the lease rules (§10). The runner's only ordinary `mc` traffic is mechanical lifecycle traffic through a **private lifecycle-only scope**: the periodic heartbeat (§10) and registering the harness's native session handle. That scope cannot read task content, perform a transition, complete a role, or dispatch work; it is not installed in the model's tool environment and is not inherited by subagents. Conversely, the role-scoped capability cannot heartbeat.

**Homie runner — one conversation, many top-level turns.** It stays warm across turns for one durable conversation and owns the conversation's automatic transport through a private **runner-scoped `mc` capability** the model never sees: poll for the next pending inbound turn belonging to its conversation, atomically claim it, resolve any permitted file-plane attachment references, supply the complete turn to the live or resumed harness session, append the completed reply through `mc` against the claimed turn, complete the claim, and continue until stopped or idled out. Claims and reply submission are fenced and idempotent, so runner death cannot create two logical replies to one inbound turn; pending work is never an in-memory queue — a fresh runner resumes from the durable claim and conversation records. The Homie agent separately receives its narrower agent-scoped `mc` surface (the §15.3 verb allowlist) for intentional Mission Control actions the model chooses to perform — which keeps chat transport out of the model's tool loop without giving an injected model access to runner transport verbs. The Homie runner never connects to Discord or the browser, chooses reply destinations, delivers outbox rows, interprets the operator's message into an action, or dispatches pipeline work.

| Responsibility | Pipeline runner | Homie runner |
| --- | --- | --- |
| Launch the configured harness through its adapter | Yes | Yes |
| Drive top-level turns | Exactly one | One at a time, many over the conversation |
| Start or resume a native harness session | Start only | Start or resume |
| Maintain the pipeline lease heartbeat | Yes, private lifecycle-scoped `mc` | No pipeline lease (§15.3) |
| Poll and claim inbound conversation turns | No | Yes, private runner-scoped `mc` |
| Submit ordinary conversational replies through `mc` | No | Yes |
| Perform role or operator actions chosen by the model | No — the agent invokes scoped `mc` directly | No — the agent invokes scoped `mc` directly |
| Dispatch work or spawn/stop containers | No | No |
| Connect to external surfaces / open the spine directly | No | No |

**Scope enforcement has two grains, and only the coarse one is kernel-backed.** The *per-container* grain is enforced: `mc` learns the run's identity — tier, role or Homie verb allowlist, fencing `run_id` — only from the read-only `run.json` at its fixed mount path, never from an argument, and the model cannot rewrite that file (§11.3). A Homie container's `mc` therefore refuses pipeline verbs, and any container's `mc` refuses another run's identity, no matter what invokes it. The *within-container* grain — the runner versus the model, sharing one uid — is best-effort by decision: the runner keeps its lifecycle and transport capabilities out of the model's tool environment, but nothing kernel-grade prevents a model from invoking them against its own run. That residue is deliberately small and already bounded by other machinery: a model that heartbeats its own lease buys nothing past the non-renewable hard deadline (§10), and a Homie model that reaches its own conversation's transport verbs is confined by the same fenced, idempotent claims that make runner death safe (§15.5). The attack that matters — a prompt-injected Homie impersonating a pipeline run's completion or another session's traffic — is closed at the per-container grain.

There are no mailbox databases, no stub `mc`, no `report.json`, and no host-side run shim at either tier: the real, scoped `mc` and the spine's own rows replace all four.

### 11.6 Heartbeats, reaping & orphans

The pipeline runner emits the heartbeat directly — `mc heartbeat <run_id>` through its private lifecycle scope (§10, §11.5). There is no heartbeat file and no mtime sensor: liveness is a spine write by the trusted runner, proving the runner and harness processes exist and nothing more; semantic progress is bounded by the lock's non-renewable hard deadline, which no heartbeat can extend.

The thresholds map cleanly onto container realities. `spawn_grace_s` absorbs the container cold path — image pull, proxy wiring, `docker run` startup — and a lock slot that was stamped but never produced a first heartbeat is a failed container spawn, reaped in seconds. The full lease timeout covers a container that heartbeated and then went silent; the hard deadline covers one that heartbeats but never finishes.

**Reaping is decide-then-effect** (Inv. 2). The tick's dispatch decision marks the run reapable and applies the §10 stale-recovery rules; the resident effects the stop — by the container's exact name, validated against a strict character set before anything is passed to a shell — because only the host side holds the container control socket. The tick evaluates both reap conditions; it never impersonates the runner or refreshes the lock. Instances run with `--rm`, so a stopped container leaves no residue to garbage-collect.

**Orphan cleanup** runs at every tick start, **scoped by the `mc-tier` label**, because the two tiers have different ground-truth liveness records. A `pipeline` container holding no live lease is stopped — the lease is exactly its liveness, and this covers the cross-Worksource seeding pass like any other leased run. A `homie` container is swept against the Homie session registry (`homie_sessions`, §15.4) and its `homie_idle_timeout`, never against the lease it deliberately does not hold. The `mc-managed` label keeps the sweep from ever touching unrelated containers on the same Docker daemon; the tier label keeps the lease-keyed rule from reaping the one lease-free tier the spec itself authorizes (§15). For the same reason, the pre-spawn at-most-one assertion checks for an existing **`mc-tier=pipeline`** container, not any `mc-managed` container — an open operator conversation must never block pipeline dispatch. The §11.1 invariant is thus precisely: **at most one *leased* container**, enforced against the pipeline tier only.

**There is no session-dir cleanup.** A session folder is never swept (Inv. 26): it holds nothing but the harness's native trace — no launch input, no terminal reports, no heartbeats, no mailboxes, no scratch — so there is nothing to age out, and every run a deployment has ever performed stays inspectable for as long as the deployment exists. This makes the file plane the system's permanent record of everything that has run and — because a harness's verbatim file can contain whatever passed through the run — the one place unredacted content lives at rest, declared in §16.1: host-local, protected by ordinary filesystem ownership; an operator who copies that storage should treat it accordingly.

### 11.7 The container runtime is a hard dependency

There is no host fallback, by design. If the container runtime is unreachable, dispatch fails closed: the tick reaps what it can, refuses to claim the lease or spawn anything, and writes a health event instead of degrading to unsandboxed execution. A stopped Docker daemon therefore stops the system — that is the accepted cost of making confinement unconditional, and runtime health is a first-class System Health signal (§13). **Docker Desktop is the supported v1 runtime, but acceptance is by capability, never by runtime identity**: `mc doctor` and onboarding run the setuid probe (a direct open on the spine refused by the kernel; the same read through `mc` succeeding) and the helper exec round-trip, and passing them is what "supported" means at runtime — the runtime's name or version string is never inspected. A probe failure refuses fail-closed with a message naming the likely causes: `no-new-privileges` forced, uid remapping (rootless mode), or Docker Desktop's Enhanced Container Isolation (which must stay off). `mc doctor` checks the whole chain: container runtime reachable and capability-probed, base image present, the mount allowlist present and parseable, gateway health, runtime-auth health per binding (§11.4), the profile's `network_allow` rules resolvable and applicable, and full profile validation.

## 12. Deployment, Supervision & Recovery

One brain, many sandboxed Worksources, one pane of glass; deployed once. Adding a venture = adding a Worksource.

```
   Native host                          │   Container runtime (one kernel = one lock domain)
   ───────────                          │   ────────────────────────────────────────────────
   resident process ──┐                 │   ┌─ spine  (the database; the message bus)
   dashboard ─────────┤  invoke real    │   │
                      │  mc across the  │   ├─ helper        ── real mc (host-originated work)
                      │  host→runtime   ├──▶│
                      │  boundary       │   ├─ agent containers ── runner ── adapter ── harness
                      │                 │   │                         └─ real mc (privileged gate)
                      └─────────────────┘   │
                                            └─ dispatch        ── wakes / reaps agents
```

Two native host units, both supervised by the host init system (Inv. 23) — macOS `launchd` or Linux `systemd` — and both stateless over the spine. On macOS both are generated as ordinary **per-user LaunchAgents** in the operator's GUI session, never LaunchDaemons: the resident's core work goes through the container runtime's socket, and the supported runtime lives in the GUI session — a pre-login daemon would outlive the very thing it needs to do its job. Losing the resident at logout costs nothing durable (every fact lives in the spine, so a logout is a longer gap between ticks); `RunAtLoad` brings both processes back at next login and dispatch resumes from the records. launchd has no dependency ordering, so the units never attempt "start after Docker" — the resident self-heals instead: retry with backoff, fail closed, write the health event. The **resident process** holds the Discord connection, runs the outbox delivery loop (§15.5), hosts the credential-injecting egress proxy (§11.4), and fires the dispatch tick — the system's one metronome (§10). It also effects every spawn and stop the dispatch decision returns (§10, §11.6), because only the host side holds the three things a spawn physically needs — the file plane, the proxy, and the container control socket. The **dashboard** is a separate native unit serving the web UI on loopback (§15.1). Neither holds authoritative state; every durable fact lives in the spine. Dispatch itself *decides* which agent to wake or reap; the resident performs the actual spawn and stop.

**Platform asymmetry.** The entire host→runtime boundary — and therefore the helper (§11.5) and the cost of host-side spine access — is an artifact of the macOS/Windows container-runtime VM. On a native Linux host the resident and dashboard share the runtime's kernel and run `mc` against the spine directly: no helper, no boundary crossing. The architecture is identical in shape on both platforms; only the cost of the host-side hop differs, and on Linux it is zero. **macOS is the primary target, decided:** the system is built and proven against the VM topology first — the helper and self-delegating `mc` (§11.5) are day-1 machinery, not portability afterthoughts — while the Linux path is kept correct by construction (the same binary simply detects it is already in the lock domain) but is not a day-1 test target.

**Recovery is a property, not a procedure.** No component holds authoritative conversational or orchestration state, so resume is automatic. If a Homie container or runner dies mid-conversation, the next inbound message makes the conversation eligible again and the next tick wakes a fresh instance that continues from the persisted turns (§10, §15.4). If the resident dies, its connections reconnect on restart and undelivered outbox rows deliver then — at-least-once with de-duplication, so nothing is lost or doubled. If the dashboard dies, it restarts, re-reads, and the browser reconnects. Across a reboot, the init system restarts both units, the spine survives in its runtime-local volume, and the first tick reaps stale leases; dispatch recomputes from the Records as it always does (§10), so no canonical state is lost and there is no index to rebuild. A flapping resident (`runner.started` repeats) is a dashboard health signal.

## 13. Observability & Metrics

Mission Control writes and maintains **no trace or observability code**, by deliberate choice. Such code would have to parse and normalize the native session formats of two independent agent harnesses (Codex and Claude Code) and track them across their versions indefinitely — an open-ended maintenance liability for a system meant to run lean and largely unattended, and one that buys nothing the harnesses do not already provide. **Observability is therefore a non-goal of the system itself: external, optional, and the operator's to run.** What the system does keep is read-only and small — the metric below, reduced over the spine, and the harnesses' own native session files, written durably into each run's mounted session folder as a matter of course (§9). The dashboard that renders these also hosts the operator's write affordances — the Console tab and the decision buttons — but every one of those travels as an `mc` verb through the native surfaces (§15); the read-only data path never participates in control flow.

**Observability is external and optional.** The harnesses already write a complete, durable record of each session to their own native files (Codex's rollout file, the Claude SDK's session JSONL), landing directly in each run's session folder and kept forever for every run, pipeline and Homie alike (§9, Inv. 26). An operator who wants to *inspect* a run installs a self-hosted **Langfuse** instance and the first-party observability plugins — `langfuse/codex-observability-plugin` for Codex and `langfuse/claude-observability-plugin` for Claude Code — and points them at those native files. That deployment is the operator's own: external to `MC_HOME`, unsupervised by Mission Control, and entirely optional. Mission Control bundles nothing, drives nothing, configures nothing (§16.3), and stores no second copy of trace content. (Install specifics — plugin names, the version floors the install cards mention like `langfuse>=4,<5`, Node 22+, Codex 0.128+, Python 3.9+, and hook behavior — are a *pointer*, not a spec contract; they move over time, so re-verify against the plugin source before relying on them.)

**Limitations of the external-observability stance** — honest edges, stated so no reader assumes the recommendation does more than it does:
- **Codex is not traced automatically under `codex exec`.** The Claude plugin's hook fires even in headless mode, so a Claude run is captured just by installing the plugin. The Codex plugin's hook fires **only in the interactive TUI, not under `codex exec`** — which is exactly how Mission Control dispatches. Capturing a Codex run would mean driving the plugin by hand against the retained rollout file, and that is custom code this decision forbids. So under a strict zero-code reading, **Claude runs trace into Langfuse automatically; Codex runs do not.** Whether a single one-shot drive command is a worthwhile exception, or Codex observability is simply unsupported, is an open operator call.
- **Every run is viewable, but as a post-mortem only.** Native files for *both* tiers are retained forever (§9, §11.6), so an operator's Langfuse can read any run — pipeline or Homie — at any later time. What no tier gets is a *live* view: plugin capture reads the native file the harness wrote, which is complete only once the session exits, so deep inspection is always after the fact.
- **No usage/cost signal.** Removing the trace layer removes the normalized token tally, so there is no in-product token or cost metric and no usage half to the routing signal — retuning rides acceptance rate alone (§9). This is a deliberate choice: a token tally would mean the adapter normalizing two usage shapes again, the very maintenance liability the decision is avoiding.
- **No in-product trace view, live or post-hoc.** Plugin-based capture was always post-hoc, and with no trace layer the dashboard shows no trace view of its own. The live operator surfaces are Board, Review Queue, Console, and System Health; deep run inspection is an out-of-band activity in the operator's separate Langfuse instance.

**Views — the tabs of the one dashboard (§15.1):** Board (kanban, one active card system-wide); Review Queue — packets as lifecycle with refinement history (`refine_streak`, saturation) + age-in-queue, carrying approve/revise/cancel as structured verbs (§15.2); Activity + separate error stream; System Health (saturation, runner/lock health w/ the heartbeat thresholds and hard deadline, backpressure occupancy, runtime-auth health, sandbox/mount denials, declared raw-TCP network allowances — shown as standing per-profile grants, not per-connection events, §11.4 — resident/dashboard liveness and outbox delivery lag); **Console** — the conversation tab, where Homie sessions are opened, attached to, and rendered (§15.4). There is no Agent Traces tab: run inspection lives in the operator's external Langfuse, not in Mission Control.

**Tracked metric** (sliced by `worksource`/`scope`/`origin`, trended):
1. **Acceptance rate** = `approved / (approved + cancelled)` over terminal decisions — read straight from the spine's terminal packet decisions written by `mc packet decide`, never from any trace or usage file, which is why it survives the trace layer's removal. The single quality verdict. Refinement bounces are tracked separately via `refine_streak` (a churn signal), never counted here. Throughput and token-utilization (maximizing consumption) remain deliberate *anti-metrics* — and there is no longer a token figure to be tempted by.

## 14. Daily Review Console

The operator's standing touchpoint — a ranked dispatch, delivered at a configured time to the configured surfaces (§15.6). At most `REVIEW_WIP_CAP` items: decision-ready packets ranked by leverage across all Worksources (Worksource, 30-second thesis, evidence, refinement history, risk/rollback, recommended decision) + batched blocked tasks awaiting input + standing state.

**Mechanism.** The Console is compiled by **Strategist(console)** — the one time-triggered dispatch in the system, riding Inv. 21's stored-schedule carve-out. A tick that finds the delivery time passed with no same-day `daily.briefing` event dispatches it ahead of pipeline selection (§10, step 0b): an ordinary leased run whose brief materializes everything it ranks — the queue, the blocked tasks, standing state — so it needs no workspace mounts at all. Its completion writes the rendered Console to `outputs/`, the `daily.briefing` activity event (which is what prevents duplicate same-day delivery), and one outbox row per configured surface (§15.5); the owning surface delivers from there. On Discord the Console arrives with decision buttons under each packet (§15.2), so the daily touchpoint is bidirectional by construction rather than by reply-parsing.

## 15. Communication Surfaces & Homies

The pipeline runs with no human in the inner loop, but the operator needs ways in — and the `mc` CLI is deliberately not one of them (§18): it is agent-facing substrate the operator never learns. There are exactly two interactive surfaces. The **dashboard** is the system's one web application — the observability tabs of §13 plus a **Console tab** for conversation; it is core, not optional. **Discord** is the optional second surface, for the operator who wants the system in their pocket. The dashboard is served by its own native unit and Discord is held by the resident (§12, §15.1), and all traffic on both reduces to one rule per direction: operator → system is always an `mc` verb, and system → operator is always an **outbox row** (§15.5). The rest of this section is the state and the interconnects that make those two rules hold.

### 15.1 The native surfaces: resident and dashboard

Two supervised native processes (§12) carry everything surface-shaped. The **resident** holds the Discord connection when that surface is enabled and runs the delivery loop that drains the outbox (§15.5). The **dashboard** serves the one web application — every tab, the read-only views and the Console alike. Both are deliberately boring: beyond their live connections they are stateless, and every durable fact they act on lives in the spine or on the file plane, reached exclusively by invoking the real `mc` (§11.5) — `mc` remains the sole spine writer (Inv. 2), and neither process runs agent logic; the resident starts and stops containers only as the mechanical hand of a dispatch decision already committed in the spine (§10). If either dies, the init system restarts it and it resumes from the registry and the outbox with nothing lost: undelivered rows are still undelivered, active sessions are still active.

There is no separate "observability web server" and no separate "web pane": one dashboard with several tabs (§13). The §13 read-only guarantee is about the observability *data path*, not these processes — the dashboard's write affordances all travel as `mc` verbs like everything else. One boundary fact governs everything downstream: **Homie never talks to Discord, the browser, or the spine directly** (§11.5). The resident is the sole holder of the Discord connection and the dashboard the sole holder of the browser connection, so the chat boundary and the credential boundary are the same boundary — even a compromised Homie can only cause its runner to submit a reply within its own conversation, and a trusted native process decides whether to deliver it.

### 15.2 Two grades of operator traffic

Everything the operator does on a surface is one of two things, and the distinction decides whether an agent is involved:

- **Structured verbs need no agent.** An approve button on the Review Queue tab, a decide button under a Discord packet alert, an unblock-with-steer form on a blocked task — each is an unambiguous one-to-one mapping onto an `mc` verb (`mc packet decide`, `mc task unblock`, …), and the owning surface process issues it directly. This is form submission, not interpretation, so it does not breach the no-agent-logic rule (§15.1). The reasons some verbs require (revise/cancel, §7) are collected by the surface — a Discord modal, a dashboard text field — before the verb fires.
- **Free text always means a Homie.** Anything that needs interpretation — shaping a new initiative, a question that requires sifting state, "scratch that" in prose — is routed into a Homie session (§15.3). The native surfaces never parse natural language; they only move it.

### 15.3 Homies

Free-text operator traffic deploys a **Homie** — a conversational session that exists for exactly one reason: the operator asked for it.

**A Homie is an extension of the operator, not a worker.** Like the read-only subagents of Inv. 14, it takes no execution lease and writes no deliverable — a conversation can last an hour, and holding the global lease would freeze all real work. Unlike them, it is top-level, multi-turn, and may issue operator verbs. It never enters producer/judge accounting (Inv. 9) and never counts against `REVIEW_WIP_CAP`: the queue protects attention the *system pushes*; a conversation is attention the operator chose to spend. Its native session file lands durably in its session folder and is kept forever, like its conversation rows (§9, §15.4). It is routed like any role: `routing.md` carries a `homie` row (default `claude-sdk`/`claude`); it sits in no producer/judge pair, so Inv. 9 places no decorrelation constraint on it.

**A Homie session is durable and resumable.** Unlike a pipeline run — ephemeral and one-shot by invariant (Inv. 5), and never resumed — a conversation is a first-class record that outlives its container. A session that idles out is `ended`, not destroyed: its registry row, conversation rows, and the harness's own native session file are kept indefinitely (§15.4), so the operator can pick the conversation back up days later and the harness continues with its full prior context. **Resumability is a Homie-tier property only;** pipeline roles need none and get none. Crucially, resume is surface-agnostic: a session started on Discord can be resumed from the dashboard and vice versa, because session identity lives in the spine, not in the connection (§15.4).

**Confinement.** Homies run containerized like everything else, under a dedicated **operator-scope profile**: read-only mounts across all Worksources' workspaces, session files, and artifacts, plus board-wide reads via `mc get/list`. This is one of two declared read-only exceptions to Inv. 22's per-Worksource scoping (the other is the Strategist(propose) seeding-scope pass, §3/§11.1) — justified because the session is filesystem-read-only and answers to the operator, whose own jurisdiction spans all Worksources by definition.

**The only write path is `mc`.** Drafting happens in conversation; nothing lands anywhere until an `mc` verb fires. The ordinary conversational reply is not even an agent verb: Homie returns its completed turn to its runner, which appends it through the runner-scoped `mc` (§11.5). Homie is the operator's intelligent console over Mission Control state — a conversational mechanism for the human to read and manually correct anything in the spine — so the verbs the *model* may issue default to **every verb whose effect lands entirely inside the spine**: all `get/list` reads, `task add` / `initiative add` (still filed `origin:user` — Homie relays operator intent), `task block/unblock`, `packet decide`, `task interrupt`, `worksource add|pause|archive`, and the Homie session-state verbs. The exclusions are structural, not trust-based — each is a verb whose effect leaves the spine, owned elsewhere: `mc dispatch` (single call site is the resident's tick, for everyone, §10); other identities' fenced verbs (`mc complete`, `mc heartbeat`, conversation claims — identity fencing, §18); outbound `reply`/outbox writes (transport scope, §11.5); and host-effect verbs a container physically cannot honor (`onboard`, `reset`). The allowlist is frozen per session at `mc homie start` and enforced by its agent-scoped `mc` capability (§11.5), with the full spine-state surface as the default — the operator can still start a deliberately narrower session. Operator actions only write state; they never dispatch (Inv. 2 untouched — the next tick picks up the result).

**Three sanctioned uses:**

1. **Shaping work before it lands.** The operator iterates with the Homie on how to decompose and sharpen a new initiative or task; when satisfied, the Homie files it through `mc` into the proposed pool as `origin:user` — at **P-1** when the operator wants it reviewed immediately (§3). The Editor still gates entry: checkable, goal-covering success criteria (Inv. 12) are non-negotiable for every task that enters the board.
2. **Asking questions** that require sifting Mission Control state, session files, or workspace files. Read-only throughout.
3. **Disrupting the flow.** `mc task interrupt <task>` is the operator's "scratch that" against an in-flight run: in one transaction the subject task is cancelled with reason `operator_interrupt` and the lease is cleared, then the resident effects the kill of the running container by name (§11.6). Partial-work recovery is deterministic and host-side, reusing the §10 stale-reap rules — the dying agent is never asked to clean up after itself. A Worker's partial work is already inert in its `mc/task-<id>` worktree (Inv. 25), so killing the container mid-write endangers nothing outside it.

### 15.4 Session state: the registry, the bindings, the conversation rows, and the resume substrate

For "talk to the same agent from Discord and from the dashboard" to be more than a slogan — and for an ended conversation to be **resumed** later from either surface — a session must be a first-class durable record, not a property of whichever connection happens to be open. Four pieces of state carry it; the first three live in the spine (§4), written only by `mc`:

- **`homie_sessions` — the registry.** One row per session: `id` (also the session's `run_id`) · `status` (`active`|`ended`|`reaped`) · `created_at` · `last_activity_at` · `container_name` · `verb_allowlist` · plus the **four locators** the spine keeps for every run's session folder (§9): where the folder is, which harness writes it (`binding` — the exact harness + model, e.g. `claude-sdk`/`claude`, captured at start), the harness's own handle for the session (`native_session_ref` — a Codex `thread_id` or a Claude SDK `session_id`, registered by the runner's lifecycle scope), and the trace's filename. The locators exist for resume: a Homie session is resumed by relaunching **the same harness** against its retained native session file (below), so the row must remember which harness wrote it, how that harness names the session to continue, and exactly where the record lives — no guessing. This is the registry §11.6's orphan sweep keys on: a `homie` container is live iff its registry row is `active`, and a row idle past `homie_idle_timeout` — measured on `last_activity_at`, which advances with every message in either direction — is `ended` by the tick, never by the lease it deliberately does not hold. **`ended` and `reaped` are not terminal for resumability:** the row, its bindings history, its conversation rows, and its native session file all persist indefinitely (§11.6), and a resume flips the row back to `active` under a fresh container. This remains the single place residency-style idle supervision exists; pipeline roles keep their per-run lifecycle (§11.1).
- **`homie_bindings` — where the conversation surfaces.** `(session_id, surface, channel_ref, bound_at)`, where `surface` is `discord`|`dashboard`|`cli` and `channel_ref` names the concrete place — a Discord channel or thread id, a dashboard connection, a terminal. A session may hold several bindings at once; that is the cross-surface requirement made into a row. A binding is created by the first traffic from a surface (or explicitly with `mc homie bind`): start a session from Discord, attach to it from the dashboard's Console tab, and both are now bound. Active bindings clear when the session ends, but the session itself stays resumable — resuming from any surface simply creates a fresh binding for that surface, which is why a Discord-born session can be resumed from the dashboard and vice versa.
- **The conversation rows — the transcript itself.** Every inbound message and reply is an append-only spine row belonging to the session's durable conversation identity, with a stable order within it and its origin surface stamped on it (Inv. 6, §15.5). The durable record and the working record are the same record: there is no separate transcript file to keep in sync and no in-memory queue that can be lost. The rows are the single rendering source — the Console tab draws from them (via `mc homie history`), any surface attaching mid-conversation or reattaching to a resumed session reads them for context — and the same records serve the Homie runner's pending-turn claims and the outbox's delivery bookkeeping, as efficient indexed queries with durable cursor/claim/delivery state rather than three copies of a message. A row carries the conversation's **text**; attachment bytes live on the file plane and travel by reference (§15.5). Conversations persist indefinitely and stay browsable forever.
- **`sessions/<id>/native.jsonl` — the resume substrate.** The harness's own native session file — Codex's `rollout-*.jsonl`, the Claude SDK's `<uuid>.jsonl` (§9) — written directly into the session's mounted folder as the session runs and retained forever (Inv. 26). Every run keeps its native file; what is special to the Homie tier is that this same file is **also the resume substrate**, not merely a post-mortem record. It is what makes resume a true *continuation* rather than a replay: re-mounting the same folder into a fresh container and relaunching the recorded harness against it brings the session back with its full internal context, not just the visible turns. It carries unredacted conversation and any tool content the session touched — part of the system's unredacted-content-at-rest store (every run's native file), called out in §16.1. The conversation rows are the fallback when a native file cannot be replayed (e.g. across a breaking harness upgrade, §11.2): a fresh session primed from the visible turns is lossier but never lock-step dependent on a file format.

Routing follows the bindings. A free-text Discord message goes to the session bound to its channel, or — if that channel's prior session has ended — resumes it; a message with no session at all starts a fresh one. The Console tab lists both active and ended sessions from the registry, attaches to any active one, and resumes any ended one. Several sessions may be active at once — each its own container, registry row, conversation, and native file; none holds the lease — and the `cli` surface participates as just another binding (`mc homie send` binds as `cli`, resuming the bound session if it had ended; it reads the conversation back with `mc homie history --follow`, needing no delivery loop) — driven in practice by a coding harness acting for the operator, never hand-typed (§18).

**Resume is a re-mount.** `mc homie resume <session> --from <surface:channel_ref>` (also reached implicitly by the routing above) reads the registry row and its locators; the resident, effecting the decision (§10), re-mounts the session's own folder into a fresh container under the recorded `binding` and starts the harness in continue-mode against `native_session_ref` — no restore, no replay, no copy. The row flips `ended → active`, a binding is created for the requesting surface, and from there the session behaves exactly like any active one — same conversation rows, same runner transport (§11.5), same outbox fan-out to every bound surface (§15.5). Pipeline roles never resume and so never re-mount; the property is the Homie tier's alone. Resume holds no lease and is not a dispatch (Inv. 2 untouched); it is the operator choosing to spend attention, like every other Homie action.

### 15.5 The message path and the outbox

**Inbound — operator to Homie.** The operator's message arrives over a websocket the resident already holds (Discord) or the dashboard already holds (browser) — realtime, push-based, no polling. That native process records it as an inbound conversation row by invoking the real `mc` (`mc homie send <session> --from <surface:channel_ref>`, running `mc homie start` first when no session is bound there): one `mc` action appends the row, stamps `last_activity_at`, and makes the turn pending for the runner. The inbound commit makes the conversation eligible to run — the resident's next timer tick wakes a Homie container if one is not already warm (§10); within an active exchange the container is already warm and the runner claims pending turns itself, so the tick latency is paid only on a session's first message or after an idle-out. The Homie runner claims the pending turn through its runner-scoped `mc`, resolves any file-plane attachment references, and supplies the complete turn to the harness (§11.5). The Discord and dashboard paths are identical except for the first hop (internet websocket versus loopback websocket) and the surface the row is tagged with.

**Outbound — Homie to operator.** Homie completes its turn and returns the response to its runner. The runner invokes the real `mc` to append the reply row and write the required destination deliveries into the outbox — a write inside the lock domain, no host boundary crossed. The native surface that owns each destination drains its outbox rows (below) and pushes over the websocket it already holds, so the final hop is a push and the reply feels immediate.

**The outbox is the one system→operator interconnect.** A spine table: `id` · `kind` (`homie_reply`|`homie_echo`|`blocked_alert`|`console`|`health`) · `session_id?` · `surface` + `channel_ref` (the concrete destination) · `payload` (small inline JSON; anything large rides as a path reference) · `created_at` · `delivered_at`. One row per (event, destination). Everything the system pushes at the operator becomes rows here — Homie replies, blocked-task alerts, the Daily Console, health events — written by `mc` inside the same transaction that produced the event, with destinations resolved at write time from the session's bindings (for conversation traffic) or the alert-class routing (§16.3, for everything else), so the surfaces stay policy-free. The owning surface drains it: `mc outbox poll`, post to the live connection, `mc outbox ack` on confirmed delivery. The outbox is delivery bookkeeping only; the reply's durable conversation row is never removed when delivery succeeds. **Delivery polling is adaptive**: the surface polls tightly for a short window right after it injects an inbound message — a reply is imminent — and falls back to lazy polling when idle, so latency is operator-imperceptible during active exchange and the cost near zero otherwise. Durability is the point — a restart re-polls and nothing pushed is ever lost.

**Attachments travel by reference.** When the operator sends an image — from Discord or the dashboard — the native surface writes the bytes onto the file plane and the inbound row *references* them; the runner reads the referenced file from its mount and supplies it with the turn. When Homie returns an image, its runner ensures the bytes land on the file plane and the reply row references them; the native surface reads the file directly and pushes it over its websocket. The bytes never enter the spine and never cross the lock-domain boundary as a payload — plain files on host-reachable storage, exactly like the traces. Nothing here is image-specific: a document, an audio clip, any modality is the same reference on the same plane, so the message bus stays uniform whatever a turn carries.

**One conversation, every bound surface.** A reply fans out to every binding the session holds, and an inbound message is echoed (`homie_echo`, with surface attribution) to every *other* bound surface — so the conversation reads identically everywhere: send from the dashboard, and the Discord thread shows both your message and the answer. The dashboard, which renders from the conversation rows, acks its rows trivially; a Discord row acks only on confirmed post.

**Warmth is latency, never correctness.** The Homie container and runner stay warm across turns within a conversation, then exit when idle — no cold start per turn, and nothing resident when no one is talking. A fresh runner resumes from the durable claim and conversation records (§11.5), so nothing depends on the warm instance surviving.

**Failure shapes.** The Homie's container exits → the resident observes it and `mc homie end --reason exited` records it. The runner dies mid-turn → fenced, idempotent claims mean no doubled reply (§11.5); traffic stops, `last_activity_at` ages out, the idle sweep ends the session (§11.6). The resident or dashboard dies → restarted by the init system, resumes from the outbox. None of these touches the lease, so no conversation failure can ever wedge the pipeline tier.

### 15.6 Push delivery: alerts and the Daily Console

The same outbox carries everything the system initiates:

- **Blocked-task alerts.** The transaction that sets `blocked` (§5) writes outbox rows per the alert-class routing. On Discord the alert carries the structured affordances of §15.2 — unblock-with-steer (a modal whose text lands in `context/<task_id>-STEER.md`), cancel — and the free-text path besides: a reply in the alert's thread starts a Homie session seeded with the blocked task's context, for the decisions that need conversation before they need a verb.
- **The Daily Review Console** (§14), compiled by Strategist(console) and delivered as outbox rows, decision buttons under each packet. The daily touchpoint is bidirectional through the same two grades of traffic as everything else.
- **Health events** (§13), routed by the same config; default to the dashboard only.

### 15.7 Operator identity

Discord is a write surface, so it is allowlisted: `[surfaces]` names the operator's Discord user ids (§16.3), and the resident ignores every event from any other author — no verb, no session, no reply. The dashboard binds to loopback by default; binding it anywhere else requires `dashboard.auth_token` and is refused without one, fail-closed. CLI traffic is the operator by definition — it originates from the operator's own shell, in practice through a coding harness acting for them (§18).

## 16. Configuration

Everything in this section is **tunable policy, never invariant** — the mechanisms of §1–§15 hold for any legal value of any knob here. The section has a second job, equally load-bearing: Mission Control is distributed as a shared open-source repository, so the configuration model must make it *structurally impossible* for an operator's particulars to land in the tracked tree.

### 16.1 The distribution boundary: repo vs. operator

The rule is a single hard line: **nothing operator-particular is ever tracked in the repository.** Operator state concentrates under one untracked root, `MC_HOME`, with exactly one host-native exception — init-system units in their native locations — written and removed only by `mc`, never by hand, and nowhere near a git tree.

**Tracked in the repo** (shared, identical for every deployment):

- `mc` source (a single static Go binary using pure-Go SQLite, `modernc.org/sqlite` — the setuid gate demands a static executable, and CGO static linking is avoidable pain, §11.5), the spine schema and migrations, the substrate triggers, the agent runner, the resident, and the dashboard (TypeScript on Bun). Toolchain versions (Go, Bun) are declared in a tracked `mise` config; the Claude Agent SDK is pinned via `package.json` plus a committed lockfile with frozen-lockfile builds.
- The frozen role directives and brief templates (Inv. 20) — these are product, not preference; an operator who edits them is forking, not configuring.
- The base image definition: Dockerfile with the two harness CLIs at pinned versions (§11.2).
- The **install script** (`install.sh`) and the **`/onboard` skill** — the guided front door of §17: the script is the deterministic setup engine, the skill the agent's instructions for running it. Both are shared automation, identical for every deployment; neither ever contains an operator's particulars.
- `config.example.toml` and `routing.example.md` — annotated templates. Onboarding copies them into the operator root; `mc` never reads them at runtime, so editing an example in the repo changes nothing for a running deployment.
- The default blocked-pattern list for mount validation (§11.3). Operators may *extend* it locally, never shrink it — the shipped list is a floor.

**Never tracked** (operator-particular, created entirely by onboarding, §17):

- **`MC_HOME`** — one directory holding the deployment's config and file-plane state: `config.toml`, `routing.md`, the mount allowlist, `sessions/` (the per-run trace-only session folders, kept forever — Inv. 26), message attachments, `outputs/`, `workflows/`, per-Worksource state and package-cache dirs, and the runtime control dirs (§5). The **spine itself is the one piece of operator state that does not live here**: it lives on a runtime-local named volume inside the container runtime (Inv. 24), written only through `mc`; its provisioning, backup, and teardown story is §16.4 (its snapshots land under `MC_HOME/backups/`). `MC_HOME` defaults to `~/.mission-control`; overridable by the `MC_HOME` environment variable, which is deliberately the only env-level knob — everything else lives in files under the root it names, so "where is my configuration" always has a one-word answer.
- **Runtime auth credentials** — captured at onboarding into the gateway's injection table and the canonical runtime control dirs (§5, §11.4), all host-side under `MC_HOME`; Worksource tool secrets live in the operator's own workspaces, outside `MC_HOME` and outside Mission Control's management (§5). `MC_HOME` therefore contains credential material and is protected by ordinary filesystem ownership, not a keychain. The other surface deserving the same care is the **native-session store** — every run's `native.jsonl` (§9), Homie and pipeline alike, kept forever (for Homie also as the resume substrate, §15.4). Each holds the raw session and whatever tool content it touched, so it can contain secret material that passed through. These files live under `MC_HOME` like everything else and are protected by the same filesystem ownership; they are not separately encrypted in v1. This is a deliberate, scoped trade — the cost of keeping every run permanently viewable and every Homie session resumable — and the native-session store is the *only* unredacted-content-at-rest surface in the system; an operator who backs up `MC_HOME` should treat it accordingly. (One further note: if an operator stands up the optional Langfuse of §13, that tool holds its own copy of trace content, which lives in the operator's Langfuse store — **outside `MC_HOME` and outside Mission Control's lifecycle** — the operator's surface to secure, not the system's. Mission Control itself drives no export and writes no second copy.)
- **Init-system units** — the launchd plists / systemd units that supervise the resident and the dashboard (§12), generated into their host-native locations.

Worksources and sandbox profiles are operator configuration too, but they are *rows in the spine*, written only through `mc` verbs — local by construction, with nothing to gitignore.

**Two fences enforce the line.** First, `mc` refuses to initialize `MC_HOME` inside any git working tree unless the chosen path is already covered by that tree's `.gitignore` — so even an operator who points the root at their checkout cannot accidentally stage their particulars. Second, the repo ships a `.gitignore` covering the conventional in-repo location (`.mc/`) as a backstop. A contributor's working clone therefore stays clean by default, not by discipline.

### 16.2 Three layers, strict precedence

1. **Shipped defaults**, compiled into `mc` — the values in the tables below. A fresh `config.toml` that sets nothing is fully valid; onboarding's "accept defaults" path is a real configuration, not a stub.
2. **Operator config** — `MC_HOME/config.toml` plus `MC_HOME/routing.md`. Machine-global; applies to every Worksource unless a profile overrides.
3. **Per-Worksource records** — `worksources`, `sandbox_profiles` rows in the spine.

Later layers override earlier ones, and no other source exists. In particular there is deliberately **no per-repository config file**: a tracked repo must never be able to change how an operator's deployment behaves, or a malicious pull request becomes a configuration vector. Unresolvable references keep the existing posture — `mc doctor` and `mc dispatch` reject and refuse to run, never fall back silently (§9.1).

### 16.3 The knobs

**Runtimes and model bindings** (`config.toml [runtimes]`; credentials are captured at onboarding into their host-side homes, §5/§17.3). This is where "which models do *I* run on" lives — the spec's own bindings are merely the shipped defaults.

| Knob | Default | Notes |
|---|---|---|
| Enabled harness families | `codex`, `claude-sdk` | provider-extensible per §9; disabling a family invalidates any routing that names it |
| Model bindings | `chatgpt`, `claude`, `minimax` (canonical ids) | each binding = `{harness, endpoint/base-url, auth (captured at onboarding, §17.3), model}`; model strings are config values verified by the live no-op turn, never compile-time constants; operators add, remove, or repoint freely |
| `runtime_auth_delivery` | tiered per binding: `gateway` for static-token bindings (`minimax`), `materialized` for OAuth bindings (`chatgpt`, `claude`) | the §11.4 decided tiering; `materialized` = the binding's canonical control dir bind-mounted (§11.4), always declared on the profile and audited, never silent |
| Forbidden-env guard list | `CODEX_API_KEY`, `ANTHROPIC_API_KEY` | metered-fallback keys (Inv. 13); `mc doctor` fails if any appears in a harness or tool env policy |

**Role routing** (`MC_HOME/routing.md`). The role → (harness, binding) map, defaulting to the §9.1 table. The values are the operator's to retune; the *mechanism* `mc doctor` enforces is fixed: every referenced harness and binding must resolve, and every producer/judge pair (`Strategist↔Editor` for proposal generation and initiative plan review; `Worker↔Verifier` for execution) must sit on decorrelated harness families (Inv. 9). An operator with different subscriptions reroutes freely inside that constraint — e.g. swapping which family produces and which judges is legal; putting Worker and Verifier on the same family is not.

**Throughput and quality budgets** (`config.toml [budgets]`):

| Knob | Default |
|---|---|
| `REVIEW_WIP_CAP` | 3 |
| `max_correction_count` | 3 |
| `dispatch_retries` (per task, infra-death budget) | 3 |
| Saturation threshold (`refine_streak ≥ n`) | 3 |

**Timing** (`config.toml [timing]`):

| Knob | Default |
|---|---|
| `timeout_minutes` / `grace_minutes` (lease) | 60 / 15 |
| `heartbeat_interval_s` / `spawn_grace_s` | 60 / 60 |
| `tick_interval_s` (heartbeat-tick timer; reap-latency bounds are threshold + one tick) | 60 |
| `homie_idle_timeout` | 30 min |
| `backup_interval_s` (spine snapshot cadence, §16.4) | 3600 |
| `backup_retain` (snapshots kept, §16.4) | 48 |
| Daily Console delivery time + IANA timezone | 08:00, operator's tz (set at onboarding) |

**Container** (`config.toml [container]`):

| Knob | Default |
|---|---|
| Container runtime | Docker Desktop — the supported v1 runtime; acceptance is by capability probe, never by runtime identity (§11.7). Operator obligations: "Start Docker Desktop when you sign in" enabled, Enhanced Container Isolation off |
| Base image tag | the pinned tag shipped with the release |
| Package-cache root | `MC_HOME/cache/<worksource>` |
| Mount allowlist path | `MC_HOME/mount-allowlist` (outside every mounted root, §11.3) |
| Additional blocked patterns | none (additive only over the shipped list) |

**Surfaces and touchpoints** (`config.toml [surfaces]`). The dashboard is core, not a toggle: its native unit always serves it, at `dashboard.bind` (default `127.0.0.1:7333`); binding beyond loopback requires `dashboard.auth_token` and is refused without one (§15.7). Discord is the optional surface (off by default): the bot token (captured at onboarding, held host-side by the resident — the sole holder of the connection), guild/channel ids, and `discord.operator_ids` — the allowlist of Discord users the resident will listen to at all (§15.7). Alert-class routing maps each push class (blocked tasks, health events, the Daily Console) to surfaces; defaults: blocked tasks and the Console to every enabled surface, health to the dashboard only (§15.6).

**Git identity** (`config.toml [git]`, per-Worksource override on the profile): the author name/email for the commits roles make inside worktrees. This must be explicit — containers run with a synthetic HOME (§11), so there is no ambient `.gitconfig` to inherit, and the operator's personal identity must never be baked into a shared image. Onboarding proposes a non-personal default (`Mission Control <mc@localhost>`).

**Observability**: there is no `[observability]` config block. Mission Control drives no trace export and runs no observability service of its own (§13), so it carries no endpoints, credentials, or knobs for one — not for OTel/Phoenix, and not for Langfuse. An operator's optional Langfuse instance (§13) is configured wherever the operator runs it, entirely outside `MC_HOME`.

### 16.4 Spine volume lifecycle

- **Naming and identity.** The volume name is derived per-deployment (`mc-spine-<short hash of the MC_HOME path>`), so dev, test, and prod deployments can never open each other's spine. First-ever provisioning writes a `meta` table — deployment UUID, schema version, created-at — and mirrors the UUID in a file under `MC_HOME`.
- **Idempotency.** Every `mc onboard` run reads `meta` before touching anything: present and current → skip; present with an older schema → migrate; absent on a non-empty volume → abort loudly. **No onboarding path may ever drop or re-initialize a spine containing data.** Destructive reset exists only as its own explicit verb (`mc reset`, confirmation required) and always takes a snapshot first.
- **Backups.** The resident's tick loop — the system's one metronome (§10) — checks elapsed time each tick and, past `backup_interval_s`, invokes `mc backup` through the warm helper: a `VACUUM INTO` onto the helper's bind-mounted `MC_HOME/backups/` path, written to a temp name and renamed on completion. Safe under Inv. 24 (a fresh copy crosses the kernel boundary, never the live locked file), consistent and non-blocking (read snapshot), and Time Machine picks the copies up for free. Extra snapshots run at resident startup and before any destructive verb. Retention keeps the last `backup_retain` snapshots, pruned by the same tick chore; "last backup time" is the newest file's mtime, no bookkeeping table. A dead resident needs no backups: nothing mutates the spine while the resident is down (Inv. 3), and host-CLI writes are covered by the startup snapshot.
- **Loss detection.** If the resident or `mc doctor` finds a fresh or empty volume where the `MC_HOME` UUID says a spine should exist, every verb **fails closed** with "spine lost — restore from backup" instead of silently re-initializing an empty brain; `mc onboard` then offers to restore the newest snapshot into a fresh volume. The same path is the runtime-migration story (e.g. switching container runtimes): snapshot, switch, restore.
- **Teardown.** Removing a deployment is `mc reset`/uninstall — snapshot first, then delete the volume; never a raw `docker volume rm` in any documented flow. Continuous WAL replication (Litestream-style) is a noted future upgrade if the ≤1h loss window ever matters.

## 17. Onboarding Flow

**The front door is one install script, not a CLI education.** A new user clones the repo and runs **`install.sh`** — either directly in a terminal, or by opening a coding harness (Claude Code) in the repo root and invoking the repo-shipped **`/onboard` skill**, whose core instruction is to run that same script and shepherd it through. The script is the Level-0 bootstrap with no setup logic of its own — prerequisite checks, build/install `mc`, hand off to `mc onboard` — and the wizard automates every deterministic step end-to-end — preflight, home scaffold, spine provisioning, image build, warm helper, supervision units — stopping to ask only where a step genuinely needs the operator: an OAuth browser flow, a subscription choice, a workspace path, a directive. Every such question is asked explicitly by the wizard and is **dual-input by construction**: answerable interactively at the terminal by the human, or supplied programmatically (flags / an answer file) by an agent that gathered the answers in conversation. The script produces the identical deployment either way.

Under the hood the script drives `mc onboard <section>` (§18) — but that is plumbing, not interface. **`mc` is an agent concept; the operator never learns it exists.** The operator's entire vocabulary is two actions: run `install.sh`, and talk to their coding harness. No operator-facing material — README, quickstart, dashboard or Discord error text — ever instructs a human to run an `mc` command; when setup breaks, the operator says so to the coding harness, which reads `mc doctor` and re-runs the failed section on their behalf. (Any `mc` verb that finds no `MC_HOME` still falls back into onboarding — a backstop for an agent that reaches for the substrate before setup, not a path an operator takes.)

The wizard's contract is unchanged by who answers it, and mirrors the system's own posture: **every step is fail-closed, every step is verified by probe rather than by assumption, and every write lands under `MC_HOME`, the spine's named volume, or the init system — never in the repo.** It is idempotent and sectioned: re-running it skips what is already healthy, and `mc onboard <section>` re-enters any single section (after a key rotation, a new subscription, a moved workspace) without touching the rest.

The sections, in order:

1. **Preflight.** Detect the OS and init system (launchd / systemd); verify the container runtime is reachable — `docker` missing → point at installing Docker Desktop; daemon down → `open -a Docker` and poll (~60s); still down → fail with the health message — and that git is present; prompt for the `MC_HOME` location and apply the §16.1 git-working-tree fence. Nothing is written until preflight passes.
2. **Home scaffold.** Create the directory tree, copy `config.example.toml` → `config.toml` and `routing.example.md` → `routing.md`, provision the spine's runtime-local named volume and initialize its schema and triggers (Inv. 24) per the §16.4 identity and idempotency rules — the derived volume name, the `meta` table, the mirrored UUID — and write the default mount allowlist. From here on the deployment exists; everything further is filling it in.
3. **Runtime auth.** For each harness family and model binding the operator enables: run the provider's own subscription auth flow (ChatGPT-plan OAuth for `codex`, Anthropic subscription auth for the `claude` binding, a Subscription key prompt for `minimax`), landing credentials in the binding's host-side home: the gateway's injection table for static-token bindings, the canonical runtime control dir for the OAuth bindings (§5, §11.4). Two checks gate each binding before it is marked configured: a **live no-op turn** through the binding's declared delivery posture (gateway injection or materialized control dir, §11.4) proves the auth actually works end-to-end, and the **forbidden-env scan** proves no metered fallback key (Inv. 13) is present in any harness or tool env policy — the wizard refuses to proceed past a binding that would silently bill.
4. **Routing.** Present the default §9.1 table against the bindings just configured. The operator accepts it or remaps roles to their own subscriptions; the wizard validates resolution and producer/judge decorrelation (§16.3) before writing, so an invalid `routing.md` can never come out of onboarding.
5. **Container.** Pull or build the base image at its pinned harness versions, start the warm helper (§11.5) and verify a host-side `mc` invocation round-trips through it, then run the **gateway probe**: a throwaway container that must (a) complete an authenticated provider call through the proxy and (b) have its direct, non-proxy egress refused. Both legs must pass — this is the §11.4 "provably wired" requirement rehearsed once at setup so the first real dispatch is not the first test.
6. **First Worksource.** An interactive `mc worksource add`: the workspace path (symlink-resolved and validated against the blocked-pattern list before acceptance), the git contract (§5 — a non-repo path offers `git init` or registers read-only; `worktree.useRelativePaths` is set on the repo, §6.2), a generated deny-by-default sandbox profile the operator reviews, the standing directive — authored with the wizard's guidance toward *checkable* success criteria, since the Editor will hold every proposal to Inv. 12 — `seeding_mode` (default `propose-only`, the right first posture for a new deployment), and the connect-time secrets advisory (§5) for any tool secrets the workspace holds.
7. **Tunables.** Accept the §16.3 defaults (recommended, and the wizard says so) or adjust them. The one knob it explains rather than just lists is `REVIEW_WIP_CAP`: it is the single number that sets the operator's daily review load, and everything else in §8 derives from it.
8. **Surfaces.** Dashboard bind address and port (defaults accepted in a keystroke — the dashboard is core and always served by its own native unit, §15.1); Daily Console delivery time and timezone; optionally Discord: the bot token (held host-side by the resident), guild/channel ids, the operator-id allowlist (§15.7), and a posted test message the operator confirms. Discord is skippable entirely — the CLI and dashboard are a complete operator surface on their own.
9. **Supervision.** Generate and load the init-system units for the resident and the dashboard (§12), then *observe one real tick fire* before declaring the section done — supervision is verified by the supervised thing happening, not by the unit file existing.
10. **Verification.** A full `mc doctor` over everything above. Optionally, `mc onboard --smoke` runs the strongest possible check: a transient Worksource (§5) carries one trivial `origin:user` task through the entire pipeline — Editor, Worker, Verifier, Packager, a real packet in the queue, an operator approve — then self-archives. Ten minutes of tokens buys proof that the whole loop, not just each part, works on this machine.

**Failure routing.** Every `mc doctor` check is tagged with the onboarding section that repairs it, and a failed check names that section in its output (`run: mc onboard runtime-auth`) — an instruction addressed to the agent shepherding the repair, never a command the operator must learn. Health degradation after setup — an expired OAuth token, a moved workspace, a stopped Docker daemon — thus routes to exactly one wizard section: the operator says "my setup is broken" to their coding harness, and the harness reads the doctor output, re-runs the named section, and asks the operator only for what the wizard's dual-input questions require.

**Upgrades.** Onboarding stamps a `config_version` into `config.toml`. A release that introduces new knobs or sections compares versions on the next `mc` invocation and runs only the delta sections, defaults pre-filled — an upgrade is a short conversation about what is new, never a re-onboarding.

## 18. `mc` Command Surface

**`mc` is agent-facing substrate, not an operator tool.** No human learns, memorizes, or hand-types this surface, and no operator-facing document names it (§17). Its invokers are the system's own components and agents: pipeline roles and Homies through their scoped capabilities, the runners through their lifecycle/transport scopes, the resident and dashboard as the mechanical hands of surface traffic, the install script during onboarding, and a coding harness (Claude Code) acting on the operator's behalf for setup, repair, and anything else shell-shaped. The operator's own surfaces are the dashboard, Discord, and conversation: the human states intent to an agent; the agent speaks `mc`.

The boundary is structural, not editorial. Every verb below binds to a scope, and scope follows from what is mounted (§11.5): the pipeline verbs (`mc complete`, `mc heartbeat`, the runner-transport verbs) require a run identity that exists only in a dispatched container's read-only `run.json`, so invoked from a host shell they are refused outright. A human who discovers the binary can reach only the host/operator scope — the same verbs their agents already run for them — never a pipeline agent's. The surface is documented here as an implementation contract, not a user manual.

`mc onboard [<section>] [--smoke]` (the §17 wizard; sections: `preflight|home|runtime-auth|routing|container|worksource|tunables|surfaces|supervision|verify`) · `mc dispatch` (the core — decides in the lock domain and returns effect data to the resident, §10) · `mc complete <task> --run <run_id> --status … [--outputs|--reason|--correction-count|--needs-operator|--infra]` (invoked directly by the completing agent through its role-scoped capability, §11.5; `--run` is the mandatory fencing token — a completion whose `run_id` no longer matches the lease's run is rejected, never double-applied; `--infra` marks a dispatch-infrastructure failure, charging `dispatch_retries` instead of any quality budget, §10) · `mc heartbeat <run_id>` (the pipeline runner's private lifecycle scope only, §11.5) · `mc packet decide <task> --approve|--revise|--cancel [--reason]` · `mc task block|unblock <task> [--reason|--steer …]` · `mc <record> get/list` · `mc worksource add|list|pause|archive` (interactive `add` is onboarding section 6, reusable standalone for every later Worksource) · `mc doctor` (validate routing, auth, sandbox profiles, schema, container runtime/images/gateway; every failure names its repairing onboarding section, §17) · `mc backup` (spine snapshot — the tick chore's verb, also manual, §16.4) · `mc reset` (destructive re-initialization: confirmation required, snapshots first, §16.4) · `mc task add` / `mc initiative add` (file human-seeded work into the proposed pool as `origin:user`; `--priority -1` expedites Editor review, §3) · `mc homie start|bind|send|reply|list|history|resume|end` (Homie sessions, bindings, and conversations, §15.4–§15.5; `resume <session> --from <surface:channel_ref>` re-mounts an ended session's folder and relaunches its recorded harness, §15.4; `reply` is submitted by the Homie runner's transport scope, never by the model; `history --follow` is the CLI's read channel over the conversation rows; `list` includes ended-but-resumable sessions) · `mc outbox poll|ack` (each native surface's delivery loop, §15.5) · `mc task interrupt <task>` (operator "scratch that", §15). Operator verbs only write state; they never dispatch — the next tick reads the result. This is the serialization boundary (Inv. 2).

**Scope partitioning is three deny rules plus fencing; everything else defaults open.** The threat model is prompt injection counterfeiting trusted records — a Worker chewing on poisoned Worksource content told to approve its own packet — not a model breaking containment, so scoping protects the *provenance* of records the operator and the resident act on, and nothing more. Exactly three restrictions are enforced, all derived from the read-only `run.json` identity (or its absence) and never from arguments:

1. **No pipeline agent gets operator verbs.** `packet decide`, `task add`/`initiative add` (they file `origin:user` — operator provenance), `task unblock`, `reset`, and `onboard` are host-scope for pipeline containers. This turns an injected "approve yourself" into a no-op instead of a merge, and keeps agent-proposed work from masquerading as operator-seeded. Homie is the deliberate exception, with its own ruled allowlist (§15.3).
2. **Writes are fenced to the caller's own identity.** Completions, verdicts, heartbeats, and conversation claims carry the caller's `run_id`/session, and mismatches are rejected — already required for correctness (zombie-run double-apply, runner-death double-reply), so the security posture is nearly free.
3. **Outbound-to-surface writes belong to the transport scope** (the Homie runner), never the model's tool environment — the outbox is an egress channel.

Everything else defaults **open**: agents may read spine records freely (`get/list` — useful context, not risk). Pipeline agents propose work only through their role-scoped verbs with agent origin — never `mc task add`.

**Role-side verbs.** The Editor's output is a *batch* of per-proposal verdicts over the whole pool, Strategist(propose) inserts N proposals under a run with no subject task, Strategist(initiative) inserts a wave in one transaction, and the Verifier writes a verdict plus a correction file — none of which fits `mc complete <task> --status`. The implementing agent designs and names these verbs as an ADR, together with the full verb-by-scope table, under three locked constraints: (a) batch outputs commit atomically — a whole verdict pass or wave lands as one transaction or not at all, never half-visible to a tick; (b) the lease's `subject` is nullable (§10) — a propose-style run holds the lease with no subject task and no placeholder rows; (c) the new verbs live in the pipeline-role scope — fenced to the calling run's identity, absent from Homie's and the host CLI's surfaces.
